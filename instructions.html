📝 Layer 1 — Soft Guardrails

Custom Instructions

Your codebase, your rules — guide Copilot with natural language. Instructions are the foundation of safe agentic development: version-controlled, composable, and always active.

What is copilot-instructions.md?

A repository-wide custom instruction file written in Markdown that automatically applies to every GitHub Copilot interaction within that repository.

✅ What It Does

  • Automatically included in every Copilot interaction
  • No explicit reference needed — always active once committed
  • Version controlled like any other source file
  • Supports coding standards, tech stack, project structure
  • Composable with path-specific and org-level instructions

⚠️ Important Caveat: "Due to the non-deterministic nature of AI, Copilot may not always follow your custom instructions in exactly the same way every time they are used." That's why instructions are soft guardrails — they guide behavior, but hooks enforce it deterministically.

The Instruction Hierarchy

When multiple instruction sources apply, Copilot combines all relevant instructions. Higher priority wins only on direct conflicts.

👤 Personal Instructions User-level, set via Settings UI — applies to all repos
▼
📂 Path-Specific Instructions .github/instructions/*.instructions.md — scoped via applyTo glob
▼
📄 Repository Instructions .github/copilot-instructions.md — applies to entire repo
▼
🤖 Agent Instructions AGENTS.md / CLAUDE.md — multi-agent compatible
▼
🏢 Organization Instructions Set by org owners in GitHub settings — applies to all repos in org
↑ Highest Priority   |   Lowest Priority ↓
💡 Key Insight: All instructions are combined and provided to Copilot — not just the highest priority. Priority only determines which instruction wins when two directly conflict.

Best Practices

GitHub's official guidance plus battle-tested community wisdom for writing effective instructions.

GitHub's 5 Tips for Better Instructions

🏷️

Elevator Pitch

Brief summary of what your app does so Copilot understands the context.

🔧

Tech Stack

Declare frameworks and technologies so Copilot uses the right APIs.

📏

Coding Guidelines

Type hints, semicolons, error handling, testing mandates.

🗂️

Project Structure

Where to find different components so the agent navigates efficiently.

📦

Available Resources

Build scripts, linters, test runners the agent can use.

✅ Do

  • Keep instructions concise — under 2 pages
  • Use bulleted lists and clear structure
  • Be specific about technologies and patterns
  • Use path-specific files for language rules
  • Update instructions as the project evolves
  • Test instructions by implementing a feature
  • Reference external docs with links

❌ Don't

  • Write lengthy documentation-style prose
  • Write ambiguous or vague statements
  • Leave technology choices open-ended
  • Put all language rules in one giant file
  • Set and forget — let them go stale
  • Assume instructions work without validation
  • Duplicate entire style guides inline
⚠️ Watch for Anti-Patterns: Token bloat — verbose instructions consume tokens for actual code context. Instruction conflicts — contradictory repo-wide and path-specific files cause non-deterministic behavior. Monolithic files — one massive file instead of composing repo-wide + path-specific. Stale instructions — references to removed technologies or outdated patterns.

Why Instructions Matter MORE in Agent Mode

When Copilot operates autonomously, instructions serve as the only proactive guidance the agent has.

Aspect 💬 Chat Mode 🤖 Agent Mode
Execution Interactive — user guides each step Autonomous — agent works independently
Scope Single question/answer Multi-file, multi-step tasks
Tool Access Limited (suggestions, explanations) Full (bash, edit, view, search, MCP)
Failure Mode User catches errors interactively Errors compound without intervention
Instruction Importance Helpful for consistency CRITICAL for safety and correctness

🤖 In Agent Mode, the Agent…

1 Explores the codebase — Instructions tell it where to look and what matters
2 Makes decisions — Instructions define which patterns and technologies to use
3 Runs commands — Instructions specify which build/test/lint tools are available
4 Creates PRs — Instructions ensure output matches team standards
5 Iterates on feedback — Instructions help it understand review expectations

Practical instruction files you can adapt for your own projects — from simple to enterprise-grade.

BASIC

Full-Stack Project Instructions

  .github/copilot-instructions.md 
## Project and code guidelines

- Always use type hints in any language which supports them
- JavaScript/TypeScript should use semicolons
- Unit tests are required, and are required to pass before PR
  - Unit tests should focus on core functionality
- End-to-end tests are required
  - End-to-end tests should focus on core functionality
  - End-to-end tests should validate accessibility
- Always follow good security practices
- Follow RESTful API design principles
- Use scripts to perform actions when available
TECH-SPECIFIC

Next.js + Prisma Stack

  .github/copilot-instructions.md 
## Tech Stack
- Frontend: Next.js 14 with App Router, TypeScript, Tailwind CSS
- Backend: Node.js with Express
- Database: PostgreSQL with Prisma ORM
- Testing: Vitest for unit tests, Playwright for E2E
- Deployment: Vercel (frontend), AWS ECS (backend)

## Coding Standards
- Use Prisma ORM for database queries (avoid raw SQL)
- Use Zustand for state management (don't use Redux)
- Use Server Components by default, Client Components only when needed
- Always specify target file names when outputting code
- Ask clear questions instead of guessing when uncertain
PATH-SPECIFIC

Scoped Instructions for Frontend & API

  .github/instructions/frontend.instructions.md 
---
applyTo: "src/components/**/*.{tsx,jsx}"
---

Use React functional components with hooks.
Always include prop type definitions.
Follow the Primer design system for UI.
Include accessibility attributes (aria-*, role).
Write component tests with React Testing Library.
  .github/instructions/api.instructions.md 
---
applyTo: "src/api/**/*.ts"
---

Use Express.js route handlers.
Always validate request body with Zod schemas.
Return standardized error responses with codes.
Include OpenAPI JSDoc comments on all endpoints.
Write integration tests for every endpoint.
AGENT MODE

Agent-Specific Safety Instructions

  .github/copilot-instructions.md (agent section) 
## Agent-Specific Guidance

### Build & Test Commands
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`
- Format: `npm run format`

### PR Conventions
- Branch naming: `feature/description` or `fix/description`
- Commit messages follow conventional commits
- All PRs require passing tests before review

### Safety Boundaries
- Never modify files in `config/production/`
- Do not delete migration files
- Always run tests before committing
- Never hardcode secrets — use environment variables

Custom Agents .github/agents/

Specialized versions of Copilot with their own persona, tool restrictions, and instructions — selected for specific tasks.

  .github/agents/readme-specialist.agent.md 
---
name: readme-specialist
description: Specialized agent for README files
tools: ['read', 'search', 'edit']
---

You are a documentation specialist focused
on README files. Your scope is limited to
documentation files only — do not modify
code files.

**Primary Focus:**
- Create and update README.md files
- Structure sections logically
- Use relative links for repo files
- Keep content under 500 KiB

**Important Limitations:**
- Do NOT modify code files
- Do NOT change generated API docs

🤖 Agent Key Facts

  • Location: .github/agents/NAME.agent.md
  • Org agents: .github-private repo
  • Frontmatter: name, description, tools, mcp-servers, hooks
  • Selection: Users pick from dropdown in IDE/CLI
  • Scoped hooks: Can define hooks directly in agent frontmatter

⚡ Agent Skills

  • Location: .github/skills/<name>/SKILL.md
  • Trigger: Automatic — Copilot selects when relevant
  • Use cases: GitHub Actions debugging, deployment procedures, release notes
  • Personal: ~/.copilot/skills/

Let Copilot Write Its Own Instructions

Use the official self-onboarding prompt to bootstrap instructions for any repository.

  Paste this into Copilot Chat (Agent Mode) 
Your task is to "onboard" this repository to a coding agent by adding a
.github/copilot-instructions.md file. It should contain information
describing how the agent, seeing the repo for the first time, can work
most efficiently.

## Goals
- Document existing project structure and tech stack.
- Ensure established practices are followed.
- Minimize bash command and build failures.

## Limitations
- Instructions must be no longer than 2 pages.
- Instructions should be broadly applicable to the entire project.

## Guidance
Ensure you include: A summary of what the app does, the tech stack
in use, coding guidelines, project structure, existing tools and
resources.
💡 Pro Tip: After Copilot generates the instructions file, review it and make edits. Then use the instructions for a real task to validate they work. Iterate until the agent produces code that matches your standards.

Getting Started

Create your first instruction file in under 5 minutes.

1

Create the file

Create .github/copilot-instructions.md in your repository root. The .github/ directory may already exist for workflows.

mkdir -p .github
touch .github/copilot-instructions.md
2

Add your project context

Start with the basics — elevator pitch, tech stack, and key conventions.

## About This Project
A customer portal for managing subscriptions
built with React and Node.js.

## Tech Stack
- Frontend: React 18, TypeScript, Primer CSS
- Backend: Node.js, Express, PostgreSQL
- Testing: Jest + React Testing Library

## Key Conventions
- Use functional components with hooks
- Always include error boundaries
- Use environment variables for config
3

Add path-specific rules (optional)

Create scoped instructions for different parts of your codebase.

mkdir -p .github/instructions
cat > .github/instructions/tests.instructions.md << 'EOF'
---
applyTo: "**/*.test.{ts,tsx}"
---
Use describe/it blocks with clear test names.
Mock external services with Jest mocks.
Test both success and error paths.
Aim for 80%+ code coverage.
EOF
4

Commit and test

Commit the file and try a real task with Copilot to validate your instructions work as expected.

git add .github/copilot-instructions.md .github/instructions/
git commit -m "Add Copilot custom instructions"
git push
5

Layer your defenses

Instructions are just the first layer. Add hooks for enforcement, configure validation tools, and set up the complete safety stack.
