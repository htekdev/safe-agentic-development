The Coding Agent

From Issue to Pull Request

Assign an issue to @copilot, and the coding agent autonomously creates a branch, writes code, runs tests, and opens a draft PR for your review.

1
Assign Task

Assign an issue to @copilot on GitHub.com, via VS Code, GitHub Mobile, or the Agents panel.

2
Autonomous Execution

Creates a branch, spins up an ephemeral dev environment, reviews context (issues, instructions, codebase), and forms an implementation plan.

3
Code & Test

Writes code, runs tests and linters, iterates on failures. Opens a draft PR tagged [WIP] for real-time tracking.

4
Developer Review

Agent updates PR with a clear title and description. Developer reviews, provides @copilot feedback for iteration, then approves and merges.

  └── Assigned to: @copilot
       └── Branch: copilot/fix-42-add-input-validation

                 ├── Commit 1: "Add email validation regex"
                 ├── Commit 2: "Add password strength check"
                 ├── Commit 3: "Add unit tests for validators"
                 ├── Commit 4: "Fix linting errors"
                 └── Status: Ready for review ✅

Configuring the Agent Environment

Place copilot-setup-steps.yml at .github/workflows/ on your default branch. This pre-configures the agent's dev environment — language runtime, dependencies, and tools.

📄 Node.js — copilot-setup-steps.yml
name: "Copilot Setup Steps"

on:
  workflow_dispatch:
  push:
    paths:
      - .github/workflows/copilot-setup-steps.yml
  pull_request:
    paths:
      - .github/workflows/copilot-setup-steps.yml

jobs:
  copilot-setup-steps:          # REQUIRED: exact job name
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
📄 Python — copilot-setup-steps.yml
name: "Copilot Setup Steps"

on:
  workflow_dispatch:
  push:
    paths:
      - .github/workflows/copilot-setup-steps.yml
  pull_request:
    paths:
      - .github/workflows/copilot-setup-steps.yml

jobs:
  copilot-setup-steps:          # REQUIRED: exact job name
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: "3.13"
          cache: "pip"
      - run: pip install -r requirements.txt

💡 Tip: Configure environment variables and secrets via Settings → Environments → copilot in your repository. The job name copilot-setup-steps is case-sensitive and required.

Agentic Workflows

Intent-Driven Automation

Write natural language instructions in markdown with YAML frontmatter. Compile to GitHub Actions with gh aw compile. Agents understand context, make decisions, and take actions on repository events.

How It Works

1
Write Markdown

Create a .md file in .github/workflows/ with YAML frontmatter defining triggers and permissions.

2
Compile

Run gh aw compile to generate a .lock.yml Actions workflow.

3
Commit & Push

Commit both the .md source and .lock.yml compiled workflow.

4
Triggered Automatically

When the configured event fires, GitHub Actions runs the agent in a sandboxed container.

Key Frontmatter Fields

onGitHub event triggers (issues, push, schedule, etc.)
permissionsGitHub token scope (read-all, or specific like issues: write)
safe-outputsAllowed write operations (add-comment, create-issue, etc.)
engineAI engine: copilot (default), claude, codex
toolsMCP toolsets available (e.g., github: toolsets: [issues])

Getting Started with gh aw

⚙️ Installation & Usage
# Install the gh-aw CLI extension
gh extension install github/gh-aw

# Compile markdown → GitHub Actions YAML
gh aw compile

# Files created:
# .github/workflows/my-workflow.md       ← your source
# .github/workflows/my-workflow.lock.yml ← compiled

# Commit both files
git add .github/workflows/my-workflow.md \
         .github/workflows/my-workflow.lock.yml
git commit -m "Add my-workflow agentic workflow"
git push

# Manual trigger for testing
gh aw run

Three ways to author:

  • GitHub Web UI — Use Copilot on github.com with prompts
  • VS Code / Coding Agent — Delegate authoring to the agent
  • Manual — Write markdown directly, compile with gh aw compile

Example Workflows

📄 issue-triage.md — Auto-Label & Triage
---
on:
  issues:
    types: [opened]
permissions:
  issues: write
tools:
  github:
    toolsets: [issues]
safe-outputs:
  add-label:
    max: 5
  add-comment:
    max: 1
---

# Issue Triage Agent

When a new issue is opened:

1. Read the issue title and body
2. Classify as: bug, feature, question,
   or documentation
3. Apply the appropriate label
4. If unclear, leave a polite comment
   asking for more details
5. If a duplicate, note the related issue
📄 activity-report.md — Daily Summary
---
on:
  schedule:
    - cron: '0 9 * * 1-5'  # Weekdays 9AM UTC
permissions: read-all
safe-outputs:
  create-issue:
    max: 1
---

# Daily Activity Report

Generate a summary of repository activity
from the past 24 hours:

- New issues opened
- Pull requests merged
- Significant discussions
- Action items for maintainers

Create an issue titled
"Daily Report: [date]" with the summary.

Security

Defense-in-Depth for Workflows

GitHub assumes agents will attempt to read/write unauthorized state, communicate over unintended channels, and be susceptible to prompt injection. The architecture is built around this threat model.

Layer 1 Planning Layer

Staged workflow with explicit data exchanges. Safe outputs pipeline specifies, limits, and sanitizes all write operations.

Layer 2 Configuration Layer

Declarative artifacts define component connections. Controls which tokens are loaded into which containers.

Layer 3 Substrate Layer

GitHub Actions runner VM with trusted containers, kernel-enforced communication boundaries, and mediated privileged operations.

🔒 Zero-Secret Agents

  • Zero access to secrets by default
  • Dedicated container with tightly controlled egress
  • Firewalled internet — only allowed destinations
  • MCP gateway in separate trusted container
  • API proxy holds LLM tokens — agent never sees them
  • chroot jail with read-only host filesystem

📋 Staged & Vetted Writes

  • All writes buffered through safe outputs MCP server
  • Workflow authors specify which writes are allowed
  • Quantity limits — e.g., max 3 PRs per run
  • Content sanitization — removes unwanted patterns
  • Only artifacts passing the full pipeline are committed

📊 Comprehensive Logging

  • Network activity logged at firewall layer
  • Model requests/responses captured by API proxy
  • Tool invocations logged by MCP gateway
  • Environment variable accesses audited
  • End-to-end forensic reconstruction

🛡️ Coding Agent Protections

  • Ephemeral, sandboxed environment — disposed after use
  • Can only push to branches it creates (copilot/*)
  • All changes via PR — cannot self-merge
  • Every step is a commit — full audit trail
  • Respects org policies, branch rules, and access controls

Guidance

When to Use Which

Choose the right tool for the job. Both systems coexist in the same repository.

🤖 Coding Agent

✅ Good fit

  • Well-scoped bug fixes with clear reproduction steps
  • Adding tests for existing functionality
  • Incremental features with clear acceptance criteria
  • Refactoring with defined scope
  • Documentation updates tied to code changes
  • Dependency updates and migrations
  • Routine technical debt cleanup

❌ Poor fit

  • Greenfield application development
  • Complex architectural decisions
  • Tasks with ambiguous requirements
  • Security-critical implementations needing expert review

⚡ Agentic Workflows

✅ Good fit

  • Automated issue triage and labeling
  • Recurring reports and summaries
  • Documentation sync on code changes
  • PR quality checks and compliance
  • Onboarding automation for new contributors

❌ Poor fit

  • One-off tasks (use coding agent instead)
  • Tasks requiring real-time human interaction
  • High-stakes deployments or infrastructure changes

Quick Start

Get Started in 5 Steps

Enable the coding agent and start assigning issues to Copilot today.

1
Verify Prerequisites

Ensure you have GitHub Copilot Pro, Pro+, Business, or Enterprise. Enable agent access in Settings → Copilot → Policies.

2
Add copilot-setup-steps.yml

Create .github/workflows/copilot-setup-steps.yml with your language runtime, dependencies, and tools.

3
Write copilot-instructions.md

Add .github/copilot-instructions.md with your project conventions, patterns, and constraints.

4
Assign Your First Issue

Write a clear issue with acceptance criteria and assign it to @copilot. Start with a simple, low-risk task.

5
Review & Iterate

Monitor the draft PR as the agent works. Use @copilot feedback to refine. Approve and merge through your normal workflow.

Tips for Best Results

📝 Write clear issues

Include acceptance criteria, file hints, and references to related issues. Agent output quality is proportional to input quality.

🎯 Scope narrowly

The agent works best on incremental, well-defined tasks. Break large features into smaller issues.

🔄 Iterate with feedback

Comment on PRs with @copilot to refine work. The feedback loop is a feature, not a workaround.

📋 Use planning mode first

For complex tasks, use Copilot's planning to convert vague requests into well-scoped issues before assigning.

🔒 Minimize safe-outputs

For agentic workflows, always define the minimum required permissions. Don't give more write access than necessary.

👁️ Review everything

The agent is a peer, not an authority. Review PRs for security, performance, architecture, and edge cases.
