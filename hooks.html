Agent Hooks & Extensions

Programmatic guardrails that intercept, validate, and govern every AI agent action — from blocking dangerous commands to enforcing test coverage.

🔒 Layer 2 — Programmatic Guardrails

Three Governance Systems

GitHub Copilot offers three complementary mechanisms for governing agent behavior at the tool-call level.

Simplest

🐚 Shell Hooks

.github/hooks/

Bash or PowerShell scripts that run on hook events. The simplest way to add guardrails — no SDK, no dependencies, fully cross-platform.

  • Write in Bash or PowerShell
  • Configured via hooks.json
  • Read JSON from stdin, write JSON to stdout
  • Stateless per invocation
Most Powerful

⚡ CLI Extensions

.github/extensions/

JavaScript/TypeScript modules using @github/copilot-sdk. Full programmatic control with session state, custom tools, and rich lifecycle hooks.

  • Stateful across the entire session
  • Can expose custom tools to the agent
  • Rich API: deny, inject context, track files
  • Import npm packages
Most Readable

📜 Hookflows

.github/hookflows/

Markdown-based workflow rules with YAML frontmatter. Declare governance policies in plain language that anyone on the team can read and review.

  • Markdown + YAML frontmatter
  • Declarative event matching
  • Block, warn, or inject context
  • No code required

🐚 Shell Hooks

The simplest governance mechanism — Bash or PowerShell scripts triggered by agent lifecycle events. Configured with a hooks.json manifest in .github/hooks/.

hooks.json Configuration

The manifest file maps event types to scripts. Each hook specifies the event it handles and the script to run.

📄 .github/hooks/hooks.json
{
  "hooks": [
    {
      "event": "preToolUse",
      "script": "block-secrets.sh"
    },
    {
      "event": "sessionStart",
      "script": "audit-log.sh"
    },
    {
      "event": "postToolUse",
      "script": "validate-output.sh"
    }
  ]
}

Hook Event Types

Event Can Block? Receives Use Case
sessionStart No Session metadata Audit logging, environment setup
userPromptSubmitted No User prompt text Prompt analysis, policy check
preToolUse ✅ YES toolName, toolArgs Block dangerous actions, validate inputs
postToolUse No toolName, toolArgs, result Validate outputs, inject context
errorOccurred No Error details Error tracking, alerting
sessionEnd No Session summary Session cleanup, final audit

Example: Block Secret File Edits

📄 .github/hooks/block-secrets.sh
#!/bin/bash
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.toolName')
ARGS=$(echo "$INPUT" | jq -r '.toolArgs')
FILE=$(echo "$ARGS" | jq -r '.path // empty')

if [[ "$FILE" =~ \.(env|pem|key)$ ]]; then
  echo '{"permissionDecision":"deny","permissionDecisionReason":"Blocked: Cannot modify secret files (.env, .pem, .key)"}'
  exit 0
fi

⚡ CLI Extensions

The most powerful governance mechanism — JavaScript/TypeScript modules using @github/copilot-sdk that maintain state across the entire agent session.

Core API

  • joinSession() — Connect to the active agent session
  • onPreToolUse — Intercept before execution can deny
  • onPostToolUse — React after execution, inject context
  • onSessionStart — Inject baseline context at startup
  • approveAll — Default permission handler

Key Capabilities

  • Stateful — Track files, patterns, and decisions across the session
  • Block actions — Return permissionDecision: "deny" to stop tool use
  • Inject context — Return additionalContext to guide the agent
  • Custom tools — Expose new tools the agent can use
  • npm ecosystem — Import any npm package

Example: Test Enforcer Extension

Tracks modified source and test files. Blocks git commit if source files were changed without corresponding test updates.

📄 .github/extensions/test-enforcer/extension.mjs
// Tracks modified source and test files across the session
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const modifiedSourceFiles = new Set();
const modifiedTestFiles = new Set();

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onPostToolUse: async (input) => {
      if (input.toolName === "edit" || input.toolName === "create") {
        const filePath = String(input.toolArgs?.path || "");
        if (/\.(test|spec)\.[jt]sx?$/.test(filePath)) {
          modifiedTestFiles.add(filePath);
        } else if (/\.[jt]sx?$/.test(filePath)) {
          modifiedSourceFiles.add(filePath);
          return {
            additionalContext:
              `[test-enforcer] Source modified: ${filePath}. Write tests before committing.`
          };
        }
      }
    },
    onPreToolUse: async (input) => {
      if (input.toolName !== "powershell") return;
      const cmd = String(input.toolArgs?.command || "");
      if (!/\bgit\b.*\bcommit\b/.test(cmd)) return;

      const untested = [...modifiedSourceFiles].filter(src => {
        const base = src.replace(/\.[^.]+$/, "");
        return ![...modifiedTestFiles].some(t => t.includes(base));
      });

      if (untested.length > 0) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason:
            `[test-enforcer] BLOCKED: Untested files:\n${untested.join("\n")}`
        };
      }
    },
  },
  tools: [],
});

📜 Hookflows

Markdown-based governance rules in .github/hookflows/. Declare policies with YAML frontmatter — no code required. The most accessible way to add guardrails for any team member.

How They Work

Each .md file in .github/hookflows/ defines a rule. The YAML frontmatter specifies the event, file pattern, and action. The markdown body provides the message shown to the agent.

  • event: file — Triggers on file create/edit
  • event: command — Triggers on shell commands
  • action: block — Prevents the action
  • action: warn — Allows but injects a warning

When to Use Hookflows

  • Protecting sensitive files from modification
  • Blocking dangerous command patterns
  • Enforcing file naming conventions
  • Adding review reminders for critical paths
  • Team-wide policies anyone can read and edit

Example: Block Environment File Edits

📄 .github/hookflows/block-env-files.md
---
name: block-env-files
event: file
on:
  paths: ['**/.env*', '**/*.pem']
  types: [create, edit]
action: block
---

# ⛔ Environment File Protection

Modifying sensitive environment files is not allowed.
Store secrets in environment variables or a secrets manager.
Use `.env.example` for documenting required variables.

Example: Warn on Workflow Changes

📄 .github/hookflows/warn-workflow-changes.md
---
name: warn-workflow-changes
event: file
on:
  paths: ['.github/workflows/**']
  types: [create, edit]
action: warn
---

# ⚠️ CI/CD Workflow Change

You are modifying a GitHub Actions workflow.
Ensure the change follows the team's CI/CD standards.
Test the workflow in a branch before merging to main.

Hook Event Lifecycle

Every agent action flows through this lifecycle. Only preToolUse can block execution.

👤 User Prompt User sends message
→
sessionStart Audit & setup
→
userPromptSubmitted Prompt analysis
→
⛔ preToolUse CAN BLOCK
→
⚙️ TOOL EXECUTES edit, powershell, etc.
→
postToolUse Validate & context
→
sessionEnd Cleanup & audit

⛔ preToolUse is the critical control point — it's the only event where hooks can return permissionDecision: "deny" to prevent an action from executing.

Getting Started

Set up your first guardrails in under 5 minutes. Choose the system that fits your team best.

Beginner

🐚 Start with Shell Hooks

  1. 1 Create .github/hooks/ directory
  2. 2 Add hooks.json with your hook definitions
  3. 3 Write a Bash script (e.g., block-secrets.sh)
  4. 4 Make it executable: chmod +x *.sh
  5. 5 Test by running a Copilot session — hooks activate automatically
Intermediate

⚡ Build CLI Extensions

  1. 1 Create .github/extensions/my-ext/
  2. 2 Run npm init and install @github/copilot-sdk
  3. 3 Create extension.mjs with joinSession()
  4. 4 Add hook handlers: onPreToolUse, onPostToolUse
  5. 5 Extensions auto-load when Copilot CLI starts in the repo
Easiest

📜 Write Hookflows

  1. 1 Create .github/hookflows/ directory
  2. 2 Add a .md file (e.g., block-env-files.md)
  3. 3 Define the rule in YAML frontmatter
  4. 4 Write the message body in markdown
  5. 5 Commit — hookflows are active immediately

Best Practices

Patterns for effective agent governance at the hook level.

✅ Do

  • Layer all three systems for defense in depth
  • Use preToolUse for hard blocks, postToolUse for guidance
  • Keep shell hooks simple — one concern per script
  • Use extensions for stateful logic (tracking files across edits)
  • Write hookflows for policies the whole team should see
  • Test hooks locally before committing

❌ Don't

  • Don't block everything — overly strict hooks slow teams down
  • Don't put complex logic in shell hooks — use extensions instead
  • Don't forget to handle the "no match" case (return nothing)
  • Don't hardcode paths — use patterns for portability
  • Don't skip the deny reason — agents need context to self-correct
  • Don't rely on a single mechanism — combine them
