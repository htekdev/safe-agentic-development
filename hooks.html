Agent Hooks & Extensions

From advisory guidance to deterministic enforcement — four systems for governing agent behavior with code.

Shell Hooks CLI Extensions Hookflows VS Code Hooks

Four Systems, Not One

GitHub Copilot has four distinct but complementary mechanisms for extending and governing agent behavior.

🐚

Shell Hooks

.github/hooks/

JSON config + shell scripts that execute at key lifecycle points. The official, documented mechanism.

Format: JSON config + bash/PowerShell
Power: Pre-tool deny, logging, validation
Support: CLI ✓ Coding Agent ✓ VS Code ✓
Best for: Simple validation, logging, CI-like checks. Start here.
⚡

CLI Extensions

.github/extensions/

Full Node.js processes using @github/copilot-sdk. Maximum power with persistent state and custom tools.

Format: ES modules via copilot-sdk
Power: Custom tools, arg mod, error recovery, state
Support: CLI ✓
Best for: Complex governance requiring persistent state & custom tools.
📋

Hookflows

.github/hookflows/

YAML workflows and Hookify rules. Declarative governance with post-lifecycle feedback — GitHub Actions for AI agents.

Format: YAML / Markdown with frontmatter
Power: Declarative rules, post-tool feedback
Support: CLI ✓ (via gh-hookflow)
Best for: Teams wanting Actions-style declarative governance.
🖥️

VS Code Hooks

.github/hooks/ + VS Code Extension API

VS Code extension API for Copilot — register custom tools, intercept agent actions, and build IDE-integrated guardrails.

Format: JSON config + TypeScript extensions
Power: Custom tools, chat participants, full IDE access
Support: VS Code ✓
Best for: IDE-integrated governance, custom tool UX, extension marketplace distribution.

🐚 Shell Hooks — .github/hooks/

JSON-configured shell commands that execute at key lifecycle points. The official, documented mechanism for both the Copilot CLI and the cloud-based Coding Agent.

How They Work

Hooks receive JSON on stdin with context about the event and return JSON on stdout to control behavior.

📥 Input (stdin): 
{
  "timestamp": 1704614400000,
  "cwd": "/path/to/repo",
  "toolName": "bash",
  "toolArgs": "{\"command\":\"rm -rf /\"}"
}
📤 Output (stdout) — Block it: 
{
  "permissionDecision": "deny",
  "permissionDecisionReason": "Destructive command blocked."
}

Configuration — hooks.json

Place at .github/hooks/hooks.json (repo-scoped) or ~/.copilot/hooks/hooks.json (user-scoped).

{
  "version": 1,
  "hooks": {
    "sessionStart": [
      {
        "type": "command",
        "bash": "echo 'Session started'",
        "powershell": "Write-Host 'Session started'",
        "timeoutSec": 10
      }
    ],
    "preToolUse": [
      {
        "type": "command",
        "bash": "./scripts/security-check.sh",
        "powershell": "./scripts/security-check.ps1",
        "cwd": ".",
        "env": { "LOG_LEVEL": "INFO" }
      }
    ]
  }
}

Hook Properties

PropertyRequiredDescription
typeYesMust be "command"
bashUnixShell command or path to bash script
powershellWindowsPowerShell command or script path
cwdNoWorking directory (relative to repo root)
envNoAdditional environment variables (key-value object)
timeoutSecNoMax execution time in seconds (default: 30)

Example: Block Secrets Hook

A shell hook that scans file writes for hardcoded secrets and blocks them before they're written.

#!/usr/bin/env bash
# .github/hooks/block-secrets.sh
# Scans preToolUse events for hardcoded secrets in file writes

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.toolName // empty')

# Only check file write operations
if [[ "$TOOL_NAME" == "create" || "$TOOL_NAME" == "edit" ]]; then
  CONTENT=$(echo "$INPUT" | jq -r '.toolArgs // empty')

  # Check for common secret patterns
  PATTERNS=(
    'ghp_[a-zA-Z0-9]{36}'           # GitHub PAT
    'ghs_[a-zA-Z0-9]{36}'           # GitHub App Token
    'AKIA[0-9A-Z]{16}'              # AWS Access Key
    'sk-[a-zA-Z0-9]{20}T3BlbkFJ'   # OpenAI Key
    '-----BEGIN.*PRIVATE KEY-----'   # Private Keys
  )

  for PATTERN in "${PATTERNS[@]}"; do
    if echo "$CONTENT" | grep -qE "$PATTERN"; then
      echo '{"permissionDecision":"deny","permissionDecisionReason":"[block-secrets] Hardcoded secret detected. Use environment variables instead."}'
      exit 0
    fi
  done
fi

# Allow by default
echo '{}'
# .github/hooks/block-secrets.ps1
# PowerShell version of the secrets scanner

$input = $Input | ConvertFrom-Json
$toolName = $input.toolName

if ($toolName -eq 'create' -or $toolName -eq 'edit') {
    $content = $input.toolArgs | ConvertTo-Json

    $patterns = @(
        'ghp_[a-zA-Z0-9]{36}',
        'ghs_[a-zA-Z0-9]{36}',
        'AKIA[0-9A-Z]{16}',
        'sk-[a-zA-Z0-9]{20}T3BlbkFJ',
        '-----BEGIN.*PRIVATE KEY-----'
    )

    foreach ($pattern in $patterns) {
        if ($content -match $pattern) {
            @{
                permissionDecision = 'deny'
                permissionDecisionReason = '[block-secrets] Hardcoded secret detected. Use environment variables.'
            } | ConvertTo-Json
            exit 0
        }
    }
}

@{} | ConvertTo-Json
{
  "version": 1,
  "hooks": {
    "preToolUse": [
      {
        "type": "command",
        "bash": ".github/hooks/block-secrets.sh",
        "powershell": ".github/hooks/block-secrets.ps1",
        "cwd": ".",
        "timeoutSec": 10
      }
    ]
  }
}
⚠️ Critical limitation: Shell hook postToolUse output is ignored by the agent — it's fire-and-forget. Only preToolUse can block. For post-tool feedback, use CLI Extensions or Hookflows.

⚡ CLI Extensions — .github/extensions/

Full Node.js processes using the @github/copilot-sdk. The most powerful governance mechanism — persistent state, custom tools, argument modification, and error recovery.

Architecture

┌─────────────────────┐      JSON-RPC / stdio       ┌──────────────────────┐
│   Copilot CLI        │ ◄──────────────────────────► │  Extension Process   │
│   (parent process)   │   tool calls, events, hooks  │  (forked child)      │
│                      │                               │                      │
│  • Discovers exts    │                               │  • Registers tools   │
│  • Forks processes   │                               │  • Registers hooks   │
│  • Routes tool calls │                               │  • Listens to events │
│  • Manages lifecycle │                               │  • Uses SDK APIs     │
└─────────────────────┘                               └──────────────────────┘

The Minimal Extension — 3 Lines of Code

No npm install, no package.json, no build step. The SDK is auto-resolved by the CLI harness.

// .github/extensions/my-ext/extension.mjs
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const session = await joinSession({
  onPermissionRequest: approveAll,
  tools: [],
  hooks: {},
});
That's it. Drop this file into .github/extensions/my-ext/extension.mjs and it's discovered automatically. Reload with /clear or extensions_reload.

onPreToolUse — The Most Powerful Hook

⛔ Deny

Block dangerous operations before execution.

return {
  permissionDecision: "deny",
  permissionDecisionReason:
    "Blocked by security policy.",
};

✅ Allow

Explicitly approve and skip permission prompts.

return {
  permissionDecision: "allow",
  additionalContext:
    "Approved by governance ext.",
};

✏️ Modify

Rewrite tool arguments before execution.

return {
  modifiedArgs: {
    ...input.toolArgs,
    command: `${input.toolArgs.command} 2>&1`,
  },
};

Three Foundational Extensions

🛡️

Repo Shield

Blocks destructive commands, detects hardcoded secrets, warns on protected file changes.

// .github/extensions/repo-shield/extension.mjs
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const DANGEROUS_COMMANDS = [
  { pattern: /rm\s+-rf\s+\/(?!\w)/i, reason: "Recursive delete from root" },
  { pattern: /DROP\s+(DATABASE|TABLE)\s/i, reason: "Destructive database operation" },
  { pattern: /git\s+push\s+.*--force\s+(origin\s+)?(main|master)/i, reason: "Force push to protected branch" },
];

const SECRET_PATTERNS = [
  { pattern: /ghp_[a-zA-Z0-9]{36}/g, type: "GitHub PAT" },
  { pattern: /sk-[a-zA-Z0-9]{20}T3BlbkFJ[a-zA-Z0-9]{20}/g, type: "OpenAI API Key" },
  { pattern: /-----BEGIN (RSA |EC )?PRIVATE KEY-----/g, type: "Private Key" },
];

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onSessionStart: async () => ({
      additionalContext: "[repo-shield] Security extension active. " +
        "Blocking destructive commands and scanning for hardcoded secrets.",
    }),
    onPreToolUse: async (input) => {
      // Block dangerous shell commands
      if (input.toolName === "powershell") {
        const cmd = String(input.toolArgs?.command || "");
        for (const { pattern, reason } of DANGEROUS_COMMANDS) {
          if (pattern.test(cmd)) {
            return {
              permissionDecision: "deny",
              permissionDecisionReason: `[repo-shield] BLOCKED: ${reason}`,
            };
          }
        }
      }
      // Scan for secrets in file writes
      if (input.toolName === "create" || input.toolName === "edit") {
        const content = String(input.toolArgs?.file_text || input.toolArgs?.new_str || "");
        for (const { pattern, type } of SECRET_PATTERNS) {
          pattern.lastIndex = 0;
          if (pattern.test(content)) {
            return {
              permissionDecision: "deny",
              permissionDecisionReason: `[repo-shield] BLOCKED: ${type} detected. Use env vars.`,
            };
          }
        }
      }
    },
  },
  tools: [],
});

Test Enforcer

Tracks modified source files and blocks commits without corresponding test changes.

// .github/extensions/test-enforcer/extension.mjs
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const modifiedSourceFiles = new Set();
const modifiedTestFiles = new Set();
const TEST_PATTERNS = [/\.test\.[jt]sx?$/, /\.spec\.[jt]sx?$/, /test_.*\.py$/, /.*_test\.go$/];

function isTestFile(path) { return TEST_PATTERNS.some(p => p.test(path)); }
function isSourceFile(path) {
  return /\.(ts|tsx|js|jsx|py|go|cs)$/.test(path)
    && !isTestFile(path) && !/node_modules/.test(path);
}

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onPostToolUse: async (input) => {
      if (input.toolName === "edit" || input.toolName === "create") {
        const filePath = String(input.toolArgs?.path || "");
        if (isTestFile(filePath)) modifiedTestFiles.add(filePath);
        else if (isSourceFile(filePath)) {
          modifiedSourceFiles.add(filePath);
          return {
            additionalContext: `[test-enforcer] Source file modified: ${filePath}. ` +
              `Write or update tests before committing.`,
          };
        }
      }
    },
    onPreToolUse: async (input) => {
      if (input.toolName !== "powershell") return;
      const cmd = String(input.toolArgs?.command || "");
      if (!/\bgit\b.*\bcommit\b/.test(cmd)) return;

      const untestedFiles = [...modifiedSourceFiles].filter(src => {
        const base = src.replace(/\.[^.]+$/, "");
        return ![...modifiedTestFiles].some(t => t.includes(base));
      });

      if (untestedFiles.length > 0) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason:
            `[test-enforcer] BLOCKED: Source files without tests:\n` +
            untestedFiles.map(f => `  - ${f}`).join("\n"),
        };
      }
    },
  },
  tools: [],
});

Lint-on-Edit

Runs the project's linter after every file edit, injecting results for immediate self-correction.

// .github/extensions/lint-on-edit/extension.mjs
import { execFile } from "node:child_process";
import { existsSync } from "node:fs";
import { resolve } from "node:path";
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

function detectLinter(filePath, projectRoot) {
  const ext = filePath.match(/\.([^.]+)$/)?.[1];
  if (["ts", "tsx", "js", "jsx"].includes(ext)) {
    if (existsSync(resolve(projectRoot, "eslint.config.mjs"))) {
      return { cmd: "npx", args: ["eslint", filePath] };
    }
  }
  if (ext === "py") return { cmd: "ruff", args: ["check", filePath] };
  return null;
}

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onPostToolUse: async (input) => {
      if (input.toolName !== "edit") return;
      const filePath = String(input.toolArgs?.path || "");
      const linter = detectLinter(filePath, process.cwd());
      if (!linter) return;

      const result = await new Promise(resolve => {
        execFile(linter.cmd, linter.args, { timeout: 30000 }, (err, stdout, stderr) => {
          resolve(err ? (stdout || stderr || err.message) : null);
        });
      });

      if (result) {
        return {
          additionalContext: `[lint-on-edit] Issues in ${filePath}:\n${result}\nFix before proceeding.`,
        };
      }
    },
  },
  tools: [],
});

📋 Hookflows — .github/hookflows/

Declarative governance workflows — like GitHub Actions but for AI pair programming sessions. Built by the community via the gh-hookflow CLI extension.

The Gap They Fill

Problem: Official postToolUse hooks are fire-and-forget — the agent ignores their output. You can log, but you can't give feedback.
Solution: Hookflows validate content after creation and force the agent to fix issues via skill injection — real post-tool feedback.

Installation

gh extension install htekdev/gh-hookflow
gh hookflow register

Hookify Rules — No Code Required

Markdown files with YAML frontmatter. Declarative, readable, and version-controlled.

---
name: block-env-files
description: Prevent creation or editing of .env files
event: file
action: block
conditions:
  - field: file_path
    operator: regex_match
    pattern: \.env
lifecycle: pre
---

Cannot create or edit `.env` files.
Use environment variables or a secrets manager instead.

Full YAML Workflow Format

For complex multi-step governance, use full YAML workflows with shell steps, expressions, and conditional logic.

name: Block Sensitive Files
description: Prevent edits to sensitive files
on:
  file:
    lifecycle: pre
    conditions:
      - field: file_path
        operator: regex_match
        pattern: \.(env|key|pem|secret)$

steps:
  - name: Deny sensitive file access
    run: |
      echo "BLOCKED: Cannot modify sensitive files"
      exit 1

🖥️ VS Code Hooks — IDE-Native Governance

VS Code's extension API provides three surfaces for hooking into Copilot: Agent Hooks (shell commands at lifecycle points), the Language Model Tool API (custom tools for agent mode), and the Chat Participant API (domain-expert assistants).

What Are VS Code Hooks?

VS Code hooks are shell commands that execute at key lifecycle points during agent sessions — the same concept as CLI shell hooks, but running inside VS Code's agent mode.

They use the same hook format as Claude Code and Copilot CLI for compatibility. Configuration lives in .github/hooks/ or is defined inline in .agent.md frontmatter.

Key distinction: VS Code hooks are shell command hooks — they spawn an external process and communicate via JSON on stdin/stdout. The VS Code Extension API (tools, participants) is a separate, more powerful surface.

Eight Lifecycle Events

VS Code supports eight hook events — a superset of the CLI's events, with extra stages for context compaction and subagent tracking.

EventWhen It Fires
SessionStartFirst prompt of a new session
UserPromptSubmitUser submits a prompt
PreToolUseBefore agent invokes any tool
PostToolUseAfter tool completes
PreCompactBefore context is compacted VS Code only
SubagentStartSubagent is spawned VS Code only
SubagentStopSubagent completes VS Code only
StopAgent session ends

Hook Configuration

Hooks are configured via JSON files in .github/hooks/. VS Code also supports UI-based configuration via /hooks chat command and AI-assisted hook generation via /create-hook.

📁 Hook file (.github/hooks/security.json): 
{
  "hooks": {
    "PreToolUse": [
      {
        "type": "command",
        "command": "node scripts/validate-tool.js",
        "windows": "node scripts\\validate-tool.js",
        "timeout": 15
      }
    ],
    "PostToolUse": [
      {
        "type": "command",
        "command": "npx prettier --write \"$TOOL_INPUT_FILE_PATH\""
      }
    ]
  }
}
⚙️ Settings (settings.json): 
// Control which hook files are loaded
"chat.hookFilesLocations": {
  ".github/hooks": true,
  "custom/hooks": true
},

// Enable hooks in .agent.md frontmatter
"chat.useCustomAgentHooks": true,

// Auto-approve tool edit operations
"chat.tools.edits.autoApprove": false
🤖 Agent-scoped hooks (.agent.md): 
---
name: "Strict Formatter"
hooks:
  PostToolUse:
    - type: command
      command: "./scripts/format.sh"
---
You are a code editing agent.

Hook I/O Protocol — Stdin/Stdout JSON

Hooks receive JSON on stdin and return JSON on stdout. Exit codes control behavior: 0 = success, 2 = blocking error, other = warning.

📥 PreToolUse Input (stdin): 
{
  "hookEventName": "PreToolUse",
  "tool_name": "editFiles",
  "tool_input": { "files": ["src/main.ts"] },
  "cwd": "/path/to/workspace",
  "sessionId": "session-123"
}
📤 PreToolUse Output (stdout) — Deny: 
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by policy",
    "additionalContext": "Read-only access"
  }
}
Permission priority (most restrictive wins): Exit code 2 > continue: false > permissionDecision: "deny" > "ask" > "allow"

VS Code Extension API — Beyond Shell Hooks

🔧

Language Model Tool API

Register custom tools that agent mode can automatically invoke. The LLM decides when to call them based on the tool description.

Define tools in package.json, register with vscode.lm.registerTool(), and implement the LanguageModelTool<T> interface. Tools include a confirmation flow for user approval.

// Define in package.json → contributes.languageModelTools
import * as vscode from 'vscode';

interface ISecurityScanParams {
  filePath: string;
}

class SecurityScanTool implements vscode.LanguageModelTool<ISecurityScanParams> {

  async prepareInvocation(
    options: vscode.LanguageModelToolInvocationPrepareOptions<ISecurityScanParams>,
    _token: vscode.CancellationToken
  ) {
    return {
      invocationMessage: `Scanning ${options.input.filePath} for vulnerabilities`,
      confirmationMessages: {
        title: 'Run security scan',
        message: new vscode.MarkdownString(
          `Scan \`${options.input.filePath}\` for security issues?`
        ),
      },
    };
  }

  async invoke(
    options: vscode.LanguageModelToolInvocationOptions<ISecurityScanParams>,
    _token: vscode.CancellationToken
  ) {
    // Run actual security analysis here
    const results = await scanFile(options.input.filePath);
    return new vscode.LanguageModelToolResult([
      new vscode.LanguageModelTextPart(results)
    ]);
  }
}

// Register in extension activate()
context.subscriptions.push(
  vscode.lm.registerTool('security_scan', new SecurityScanTool())
);

Chat Participant API

Create @-mentionable domain experts in VS Code Chat. Participants own the conversation and orchestrate tasks themselves.

import * as vscode from 'vscode';

export function activate(context: vscode.ExtensionContext) {
  const reviewer = vscode.chat.createChatParticipant(
    'security.reviewer',
    handleRequest
  );
  reviewer.iconPath = vscode.Uri.joinPath(context.extensionUri, 'shield.png');
}

const handleRequest: vscode.ChatRequestHandler = async (
  request, context, stream, token
) => {
  // User typed: @security-reviewer check this file
  stream.markdown('🔍 Scanning for security issues...\n\n');

  // Access VS Code APIs: open files, run diagnostics, etc.
  const editor = vscode.window.activeTextEditor;
  if (editor) {
    const doc = editor.document;
    stream.markdown(`Reviewing \`${doc.fileName}\`...\n`);
    // Perform security analysis, stream results back
  }

  return { metadata: { command: 'review' } };
};
Tools vs. Participants: Tools are called by the LLM automatically during agent mode. Participants own the conversation when a user types @name. Tools for automated governance; participants for interactive review.

Best Practices

When to use each system and common governance patterns.

🐚 Start With Shell Hooks When…

  • You need simple pre-tool validation
  • You want logging and audit trails
  • You need cloud Coding Agent support
  • Your team uses diverse languages (not just Node.js)
  • You want the officially documented path

⚡ Graduate to Extensions When…

  • You need persistent state across hook calls
  • You want custom tools registered in the agent
  • You need to modify arguments or results
  • You want post-tool feedback that the agent sees
  • You need automatic error recovery

📋 Add Hookflows When…

  • You want declarative, YAML-based governance
  • Your team prefers GitHub Actions-style workflows
  • You need post-tool feedback without writing JS
  • You want governance rules that non-developers can read
  • You need git-push governance (pre/post push)

🖥️ Use VS Code Hooks When…

  • You want IDE-integrated governance with native UI
  • You need custom tools with confirmation dialogs
  • You want to distribute governance via the marketplace
  • You need access to VS Code APIs (editors, diagnostics, terminals)
  • You want agent-scoped hooks in .agent.md files

The Governance Stack — Layers Build on Each Other

Layer 5: Hookflows Declarative YAML rules, post-lifecycle feedback
Layer 4: VS Code Hooks IDE-native hooks, custom tools, chat participants
Layer 3: CLI Extensions Programmatic Node.js hooks with full SDK access
Layer 2: Shell Hooks JSON-configured scripts at lifecycle points
Layer 1: Copilot Defaults Built-in safety — permission prompts, network firewalls, sandboxing

Common Governance Patterns

Governance NeedHook Solution
Block destructive commandsonPreToolUse → deny rm -rf /, DROP DATABASE, force push
Prevent secret leaksonPreToolUse → scan file writes for API keys, tokens, passwords
Enforce testingonPostToolUse → track changes; onPreToolUse → block untested commits
Code qualityonPostToolUse → run linters after every edit, inject feedback
Audit trailonPreToolUse + onPostToolUse → log all tool calls
Protected filesonPreToolUse → warn/block edits to .env, Dockerfile, terraform/
Session policiesonSessionStart → inject org-wide coding standards
Error resilienceonErrorOccurred → auto-retry recoverable failures

Getting Started — Your First Hook in 5 Minutes

Choose your path. Shell hooks for simplicity, extensions for power.

🐚 Path A: Shell Hook

  1. Create the hooks directory 
    mkdir -p .github/hooks
  2. Write hooks.json 
    {
  "version": 1,
  "hooks": {
    "preToolUse": [{
      "type": "command",
      "bash": ".github/hooks/guard.sh"
    }]
  }
}
  3. Write the hook script 
    #!/usr/bin/env bash
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.toolName')
if [ "$TOOL" = "powershell" ]; then
  CMD=$(echo "$INPUT" | jq -r '.toolArgs.command // ""')
  if echo "$CMD" | grep -qE 'rm\s+-rf\s+/'; then
    echo '{"permissionDecision":"deny","permissionDecisionReason":"Blocked!"}'
    exit 0
  fi
fi
echo '{}'
  4. Make it executable & commit 
    chmod +x .github/hooks/guard.sh
git add .github/hooks/ && git commit -m "Add security hook"

⚡ Path B: CLI Extension

  1. Create the extensions directory 
    mkdir -p .github/extensions/my-guard
  2. Write extension.mjs 
    import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onSessionStart: async () => ({
      additionalContext: "Guard active!",
    }),
    onPreToolUse: async (input) => {
      if (input.toolName === "powershell") {
        const cmd = String(input.toolArgs?.command || "");
        if (/rm\s+-rf\s+\//.test(cmd)) {
          return {
            permissionDecision: "deny",
            permissionDecisionReason: "Blocked!",
          };
        }
      }
    },
  },
  tools: [],
});
  3. That's it — no install needed

    The SDK is auto-resolved. No npm install, no package.json, no build step. The extension is discovered and loaded automatically.

  4. Reload mid-session 
    /clear
# or use extensions_reload tool

Deployment Scopes

📁 Repository (Highest Priority)

.github/hooks/hooks.json
.github/extensions/*/extension.mjs

👤 User / Personal

~/.copilot/hooks/hooks.json
~/.copilot/extensions/*/extension.mjs

⚙️ Key Rules

Project extensions shadow user extensions. Entry point must be extension.mjs. Extensions run as isolated child processes.
