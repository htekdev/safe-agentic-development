Planning Mode & Spec-Driven Development

Specify → Plan → Implement → Verify — intent is the new source of truth. Stop vibe-coding and start building with clarity, structure, and predictable outcomes.

The Foundation

Plan Before You Code

Planning Mode is a plan-before-code approach built into GitHub Copilot. Instead of immediately generating code, Copilot first creates a structured implementation plan you review, refine, and approve.

📋

Planning Mode (Built-in Feature)

A built-in Copilot feature for generating step-by-step implementation plans within a session. Use /plan or Shift+Tab to activate. Creates structured plans with checkboxes, verification steps, and implementation order.

Available in VS Code, Visual Studio, JetBrains, Eclipse, Xcode, and Copilot CLI

📐

Spec-Driven Development (Methodology)

A broader methodology where Markdown specifications serve as the requirements documentation and the primary instruction set for AI agents. Uses a hierarchy of spec files — constitution, spec, plan, and tasks — as the source of truth.

Works with GitHub Copilot, Claude Code, Gemini CLI, and more

The Core Loop

Plan → Implement → Verify

The fundamental rhythm of planning mode — a self-correcting loop that ensures quality at every step.

Design architecture
Define steps

→

⚡ IMPLEMENT

Code changes

✅ VERIFY

Run tests
Lint & build

🔄 SELF-CORRECT

Fix errors
Adjust approach
Iterate

How the Agent Executes the Loop

  1. Analyze context — Read codebase, understand requirements, resolve ambiguity
  2. Propose edits — Suggest code changes and terminal commands aligned to the plan
  3. Monitor outputs — Watch for compile errors, test failures, and lint warnings
  4. Self-correct — Automatically fix issues, adjust approach, and re-verify
  5. Repeat — Continue until all plan steps are complete and verified
Getting Started

How to Use Planning Mode

Two workflows — pick the one that fits your environment.

🟣 In VS Code

  1. Open Chat — Press Ctrl+Alt+I and select Plan from the agents dropdown
  2. Describe your task — Enter a high-level goal with context
  3. Research phase — The Plan agent analyzes your codebase and asks clarifying questions
  4. Review the plan — Get a structured plan with checkboxes and verification steps
  5. Iterate — Refine through follow-up prompts until the plan is right
  6. Execute — Choose "Start Implementation" or "Open in Editor"
// In VS Code Chat (Plan agent selected)
/plan Implement user authentication with OAuth2 and JWT.
Use our existing Express middleware in src/middleware/auth.ts.
Follow the repository pattern in src/repositories/.
Include integration tests with our Vitest setup.

Settings: chat.planAgent.defaultModel — select planning model
github.copilot.chat.implementAgent.model — select implementation model

🟢 In Copilot CLI

  1. Toggle plan mode — Press Shift+Tab or use the /plan command
  2. Copilot analyzes — Researches your codebase and asks clarifying questions
  3. Review plan.md — Structured plan with checkboxes is saved to session state
  4. Edit the plan — Press Ctrl+Y to open in your default editor
  5. Approve — Confirm the plan and let the agent implement step-by-step
  6. Verify — Agent runs tests, lint, and build automatically
# One-off plan command
$ copilot
> /plan Add OAuth2 authentication with Google and GitHub providers

# Or toggle plan mode for the session
> Shift+Tab # Toggles plan mode on/off

# View and manage your plan
> /session plan # View current plan
> Ctrl+Y # Edit in external editor

When to Use Plan Mode

✅ Use Plan Mode For

  • • Complex multi-file changes
  • • Refactoring with many touch points
  • • New features touching multiple modules
  • • Architecture decisions that matter
  • • Greenfield project setup

❌ Skip Plan Mode For

  • • Small single-file fixes
  • • Simple variable renaming
  • • Local, contained bug fixes
  • • Single configuration changes
  • • Quick documentation edits
The Methodology

Spec-Driven Development

A four-phase workflow where Markdown specifications serve as both requirements documentation and the primary instruction set for AI agents.

"SDD is not about writing exhaustive, dry requirements documents. It's about making your technical decisions explicit, reviewable, and evolvable. Think of it as version control for your thinking." — Den Delimarsky, Microsoft for Developers, September 2025
⚙
constitution.md

Non-Negotiable Principles

Foundation layer — opinionated stack decisions, coding standards, architectural constraints. Set once, referenced by every phase.

1
/specify → spec.md

What & Why

Define the problem, user stories, acceptance criteria, and scope — without technical decisions. Focus on motivations, stakeholders, and success criteria.

# spec.md — User Profile Editing ## Problem Statement Users cannot update their profile information after account creation. ## User Stories - As a user, I can update my display name and bio - As a user, I can upload a new avatar (max 2MB, JPEG/PNG) - As a user, I can change my email with verification ## Acceptance Criteria - [ ] Profile changes save within 2 seconds - [ ] Invalid uploads show clear error messages - [ ] Email changes require clicking a verification link
2
/plan → plan.md

How

Define the technical architecture — frameworks, libraries, data models, API contracts, and integration points. Grounded by the constitution.

# plan.md — User Profile Editing ## Architecture - Express.js REST API with existing JWT auth middleware - Multer for file uploads, Sharp for image processing - PostgreSQL with existing Knex.js migrations ## API Design - PATCH /api/users/:id — Update name, bio - POST /api/users/:id/avatar — Upload avatar - POST /api/users/:id/email — Request email change ## Data Model Changes - Add avatar_url (VARCHAR 512) to users table - Create email_verifications table
3
/tasks → tasks.md

Executable Work

Break spec + plan into small, reviewable, dependency-ordered tasks. Each task is implementable and testable in isolation — like TDD for your AI agent.

# tasks.md — User Profile Editing ## Phase 1: Data Layer - [ ] Add avatar_url column migration - [ ] Create email_verifications table migration - [ ] Update UserRepository with profile update methods - [ ] Write unit tests for UserRepository changes ## Phase 2: API Layer - [ ] Create PATCH /api/users/:id endpoint - [ ] Create POST /api/users/:id/avatar endpoint - [ ] Add Multer middleware for file uploads - [ ] Write integration tests for new endpoints ## Phase 3: Email Verification - [ ] Create email verification service - [ ] Add POST /api/users/:id/email endpoint - [ ] Write tests for email change flow
4
Agent Execution

Implement → Test → Verify → Commit

The agent tackles tasks one by one. Instead of reviewing thousand-line code dumps, you review focused changes. The agent knows what (spec), how (plan), and what to work on (tasks). Your role: verify at each checkpoint.

Tooling

GitHub Spec Kit

An open-source toolkit from GitHub that scaffolds the complete SDD workflow — works with Copilot, Claude Code, Gemini CLI, and more.

Install Spec Kit

# Install and initialize Spec Kit
$ uvx --from git+https://github.com/github/spec-kit.git \
    specify init my-project

Creates the full SDD scaffolding with templates, prompt files, and helper scripts — ready to use in 30 seconds.

Anatomy of a Plan

What a Good plan.md Looks Like

A well-structured plan gives the agent everything it needs — context, steps, verification criteria, and dependency order.

📄 plan.md — Generated by /plan
# Implementation Plan: User Profile Editing ## Overview Add profile editing capabilities to the Express.js API, including name/bio updates, avatar upload, and email change with verification. ## Constraints - Use existing JWT auth middleware (src/middleware/auth.ts) - Follow repository pattern (src/repositories/) - Avatar max size: 2MB, JPEG/PNG only - All changes logged to activity audit table ## Implementation Steps ### Phase 1: Data Layer - [ ] Create migration: add avatar_url to users - [ ] Create migration: email_verifications table - [ ] Update UserRepository with CRUD methods - [ ] Write unit tests for repository changes ### Phase 2: API Endpoints - [ ] PATCH /api/users/:id — update name, bio - [ ] POST /api/users/:id/avatar — upload avatar - [ ] POST /api/users/:id/email — request change - [ ] Add input validation middleware - [ ] Write integration tests ### Phase 3: Email Verification - [ ] Implement verification token generation - [ ] Create GET /api/verify-email/:token - [ ] Send verification emails via SendGrid - [ ] Write end-to-end tests ## Verification - [ ] All tests pass (npm test) - [ ] Lint passes (npm run lint) - [ ] Build succeeds (npm run build) - [ ] Manual test: update profile via API - [ ] Manual test: avatar upload flow

Key Principles

  • ✓ Clear problem statement — What problem does this solve?
  • ✓ Phased implementation — Backend → API → Frontend, ordered by dependency
  • ✓ Checkboxes for tracking — The agent updates status as it executes
  • ✓ Explicit constraints — Existing patterns, limits, and requirements
  • ✓ Verification steps — What "done" looks like, with concrete commands
  • ✓ Testable in isolation — Each task produces a reviewable, focused change

The Four Modes of Copilot

📋

Plan Mode

Clarify intent and structure before writing code

↳ Complex features, architectural work
⚡

Agent Mode

Implement incrementally with autonomous execution

↳ After planning, for step-by-step execution
✏️

Edit Mode

Refine and polish code with targeted changes

↳ Small adjustments, formatting, polish
💬

Ask Mode

Review, explain, or question decisions

↳ Understanding code, Q&A, exploration

How Planning Feeds Agent Execution

  1. Plan — Creates structured plan.md with implementation steps
  2. Agent reads plan — Agent mode picks up the plan and executes step-by-step
  3. Session memory — Plan persists in session state for continuous reference
  4. Verification — Agent runs tests and builds after each step automatically
  5. Self-correction — Agent reads errors, adjusts approach, and retries

Execution Options

VS Code — Interactive

"Start Implementation" in the current session with real-time oversight

CLI — Autopilot

Agent executes the plan autonomously while you monitor or work on other tasks

Coding Agent — Cloud

Assign an issue to Copilot — it creates a branch, implements, runs tests, and opens a PR

CLI — /delegate

Tips & Patterns

Best Practices for Effective Planning

Patterns that lead to better plans, faster execution, and more predictable results.

✅ DO

  • • Start with a clear problem statement
  • • Break into phases (backend → API → frontend)
  • • Include explicit verification steps
  • • Reference existing code patterns in your codebase
  • • Include constraints and non-functional requirements
  • • Make each task testable in isolation

❌ DON'T

  • • Skip straight to implementation steps
  • • Leave plans vague ("implement the feature")
  • • Ignore dependency ordering between tasks
  • • Forget rollback and error handling steps
  • • Create thousand-line monolithic tasks
  • • Skip the "specify" phase and jump to code

High-Quality Planning Prompt

Detailed first prompts produce better plans. Include context, constraints, and existing patterns.

/plan Implement user profile editing for our Express.js API.

Requirements:
- Authenticated users can update name, bio, and avatar
- Avatar upload limited to 2MB, JPEG/PNG only
- Email changes require verification
- All changes audited in activity log

Existing patterns:
- Use existing JWT auth middleware in src/middleware/auth.ts
- Follow the repository pattern in src/repositories/
- Include integration tests using our Vitest setup

Constraints:
- PostgreSQL with Knex.js migrations
- Must work with existing CI pipeline (lint + test + build)
- Follow REST API conventions in existing endpoints
🌱

Greenfield (0→1)

New projects benefit most from upfront specs — prevents the agent from producing generic, off-brand solutions. Define your stack, patterns, and principles before writing a single line.

🔧

Feature Work (N→N+1)

Adding to existing systems — specs force clarity on integration points, data contracts, and backward compatibility. Reference existing patterns explicitly.

🏗️

Legacy Modernization

Capture business logic in specs first, then rebuild without inheriting tech debt. The spec preserves intent while the implementation gets a clean slate.
