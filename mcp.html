Model Context Protocol

The universal open standard for connecting AI agents to tools, data sources, and APIs — one protocol to rule them all.

The N×M Problem, Solved

MCP is an open-source standard introduced by Anthropic that provides a universal interface for AI models to connect with external systems — like USB-C for AI.

❌ Before MCP

M AI apps × N data sources =
M × N custom integrations

Copilot→Slack
Claude→DB

✅ With MCP

M clients + N servers =
M + N standard connections

Copilot ↔ MCP ↔ Any Server
Claude ↔ MCP ↔ Any Server
ChatGPT ↔ MCP ↔ Any Server

"Think of MCP like a USB-C port for AI applications. Just as USB-C provides a standardized way to connect electronic devices, MCP provides a standardized way to connect AI applications to external systems."

How It Works

  1. Client connects to configured MCP servers at startup
  2. Client performs capability discovery — asking what tools and resources each server offers
  3. Discovered capabilities are registered for the AI to use during conversations
  4. When the AI needs data or action, it requests it from the appropriate server
  5. Server retrieves data or executes actions and returns results via the standardized interface

Three Core Primitives

MCP servers expose three types of capabilities to AI clients.

⚡

Tools

Functions the AI can invoke to perform actions. The AI decides when and how to call them based on context.

create_issue run_query search_code take_screenshot
📄

Resources

Data sources the AI can read from. Application-controlled — the host decides when to include them.

File contents DB records API responses Live data
💬

Prompts

Pre-defined templates for common workflows. User-controlled — invoked explicitly by the user.

Code review Analysis Summarization Triage

⚠️ Note: The Copilot coding agent currently only supports Tools from MCP servers — not Resources or Prompts.

Transport Types

MCP supports two active transport mechanisms for client-server communication.

💻

STDIO

Local Recommended

Server runs as a child process using standard input/output streams.

Host App
→ stdin/stdout →
MCP Server
✅ Simple setup, no network config
✅ Very low latency
✅ Inherits local user permissions
📌 Best for: dev tools, CLI, IDE integrations
⚠️ Limited to local machine, single user
🌐

Streamable HTTP

Remote Scalable

Server runs as an HTTP endpoint with optional SSE for streaming responses.

Client
→ HTTP + SSE →
Remote Server
✅ Network-accessible, multi-user
✅ Supports OAuth 2.1 authentication
✅ Scalable, centrally managed
📌 Best for: shared tools, cloud, production
⚠️ More complex setup, requires TLS

Build Your Own

Create custom MCP servers with official SDKs in TypeScript, Python, C#, Go, Java, Kotlin, Ruby, and Swift.

TypeScript Python C# / .NET Go Java Kotlin Ruby Swift
📄 Minimal TypeScript MCP Server
import { Server } from "@modelcontextprotocol/sdk/server/index.js"; import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js"; import { CallToolRequestSchema, ListToolsRequestSchema, } from "@modelcontextprotocol/sdk/types.js"; import z from "zod"; // Create the server const server = new Server( { name: "my-server", version: "1.0.0" }, { capabilities: { tools: {} } } ); // Define available tools server.setRequestHandler(ListToolsRequestSchema, async () => ({ tools: [{ name: "greet", description: "Greet a user by name", inputSchema: { type: "object", properties: { name: { type: "string", description: "User's name" } }, required: ["name"], }, }], })); // Handle tool invocations server.setRequestHandler(CallToolRequestSchema, async (request) => { if (request.params.name === "greet") { const { name } = z .object({ name: z.string() }) .parse(request.params.arguments); return { content: [{ type: "text", text: `Hello, ${name}!` }], }; } return { isError: true, content: [{ type: "text", text: `Unknown tool: ${request.params.name}` }], }; }); // Start the server const transport = new StdioServerTransport(); await server.connect(transport);

🛡️ Input Validation

Use Zod (TS), Pydantic (Python), or equivalent to validate all tool inputs before processing.

⚠️ Error Handling

Wrap handlers in try-catch. Return structured errors with isError: true.

📡 Discovery

Register all tools with clear names, descriptions, and input schemas for the AI to understand.

🔄 Async Patterns

Use async/await for all external API calls and I/O operations.

🪵 Logging

Log errors to stderr — stdout is reserved for MCP protocol communication in STDIO mode.

🔌 Transport Choice

Use STDIO for local/CLI tools. Use Streamable HTTP for remote/shared/production servers.

Enterprise Governance Patterns

MCP provides enterprises with a standardized, governable way to extend AI agent capabilities.

🏢

Centralized Tool Management

Administrators configure which MCP servers are available at the repository, organization, or user level. Add capabilities incrementally without modifying the core AI system.

🔐

Permission Boundaries

MCP servers operate within defined permission boundaries. The GitHub MCP Server's default token has read-only access to the current repository only.

📋

Audit Trail

All tool invocations are logged and traceable, providing compliance-grade visibility into what the AI agent accessed and modified.

👁️

Read-Only Mode

Enable read-only mode for safe exploration. The agent can query and analyze without any write operations.

🔒 Read-Only Mode
{ "servers": { "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/", "headers": { "X-MCP-Readonly": "true" } } } }
🎯 Tool Scoping
{ "servers": { "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/", "tools": [ "listPullRequests", "getPullRequest", "searchPullRequests" ] } } }

MCP Security Best Practices

Securing MCP in production requires attention to authentication, permissions, and isolation.

🔑

Authentication (OAuth 2.1)

MCP servers follow strict OAuth authorization flows with explicit user consent per scope.

  • Short-lived, minimally scoped tokens
  • Cryptographically secure session IDs
  • HTTPS required in production
  • Validate token audiences
🛡️

Permission Scoping

Follow the principle of least privilege — only expose what the server needs.

  • Read-only modes by default
  • Tool whitelisting per server
  • Per-user permission enforcement
  • Limit exposed tool surface area
🚫

Prompt Injection Prevention

Treat all data from clients or LLMs as untrusted input.

  • Validate input lengths, types, patterns
  • Use allow-lists for commands and paths
  • Centralize validation logic
  • Fewer tools = smaller attack surface
🔒

Transport Security

Encrypt all communication beyond localhost with TLS.

  • TLS encryption for all remote connections
  • Mutual authentication (client certs)
  • SSRF mitigations for OAuth URLs
  • Block private IP range requests
🗝️

Secret Management

Never store credentials in configuration files — use secret vaults.

  • Hardened vault for all credentials
  • Automated secret scanning
  • Runtime credential usage monitoring
  • Use COPILOT_MCP_ prefix for agent
📦

Sandboxing & Isolation

Run MCP servers in sandboxed, containerized environments.

  • Containerized execution environments
  • Minimal default privileges
  • OS-level access controls (SELinux)
  • CPU, memory, and network limits

⚠️ GitHub's Official Guidance

  • Enabling third-party MCP servers may impact agent performance and output quality — review thoroughly
  • By default, the coding agent does not have write access to MCP tools — review available tools carefully
  • Update the tools field with only the necessary tooling
  • Carefully review configured MCP servers prior to saving

MCP vs Traditional API Integrations

Why MCP is a better fit for agent-based workflows compared to custom REST/GraphQL integrations.

Aspect Traditional APIs MCP
Integration Model Custom code per AI + data source (M×N) Standardized protocol (M+N)
Discovery Manual — developer reads API docs Automatic at runtime
Schema Varies (REST, GraphQL, gRPC) Standardized JSON-RPC 2.0
Agent Awareness Must be trained per API Dynamic capability negotiation
Reusability Built for one AI application Works across all MCP clients
Maintenance Update each integration independently Update server once, all clients benefit
Auth Custom per API Standardized OAuth 2.1
