THE STORY

What Are GitHub Agentic Workflows?

A new paradigm for repository automation — from YAML rules to natural language intent.

GitHub Agentic Workflows (gh-aw) are AI-powered automations that run coding agents — Copilot, Claude, or Codex — inside GitHub Actions to perform complex, multi-step repository tasks from natural language instructions written in Markdown.

Announced: February 13, 2026 (Technical Preview)
Created by: Don Syme (GitHub Next) and Peli de Halleux (Microsoft Research)
License: MIT (open source)
Repository: github/gh-aw

Instead of complex GitHub Actions YAML with deterministic step logic, you write workflows in plain Markdown. YAML frontmatter defines triggers, permissions, and tools. The Markdown body contains natural language instructions for the AI agent. The gh aw compile command transforms this into a security-hardened GitHub Actions workflow.

The Vision: Continuous AI

Continuous AI is the systematic, automated application of AI to software collaboration. Not a replacement for CI/CD, but an augmentation — extending automation to subjective, repetitive tasks that deterministic pipelines struggle to express.

"Software engineering has always included work that's repetitive, necessary, and historically hard to automate... Continuous AI automates tasks requiring reasoning — work that cannot be expressed as rules or flowcharts."
— GitHub Blog: Continuous AI in Practice
💡 The Paradigm Shift: CI/CD handles tasks expressible as rules. Continuous AI handles tasks expressible as intent — "ensure documentation matches code behavior," "triage this issue based on project context."
"Imagine visiting your repository in the morning and feeling calm because you see: Issues triaged and labelled, CI failures investigated with proposed fixes, documentation updated to reflect recent code changes, two new pull requests that improve testing await your review. All of it visible, inspectable, and operating within the boundaries you've defined."
— Don Syme & Peli de Halleux, GitHub Blog

Markdown-First Authoring

Workflows are authored as .md files in .github/workflows/. YAML frontmatter defines configuration; the Markdown body contains natural language instructions.

---
on:
  issues:
    types: [opened]
permissions: read-all

  add-comment:

# Issue Clarifier

Analyze the current issue and ask for
additional details if the issue is unclear.

The Compilation Pipeline

The gh aw compile command transforms .md into a security-hardened .lock.yml through five phases:

1
Parsing & Validation
Extract frontmatter, validate JSON schema, resolve imports
2
Job Construction
Generate isolated jobs with distinct trust boundaries
3
Action SHA Pinning
Resolve every action to full commit SHA — defense against tag-hijacking
4
Security Scanners
actionlint, zizmor, poutine — linting and vulnerability detection
5
Strict Mode
No write permissions in agent, explicit network config, no wildcards

Multi-Job Execution Pipeline

The compiled .lock.yml executes as a strict sequence of isolated jobs:

JobPermissionsPurpose
pre_activationMinimalRole checks, stop-after deadlines, skip-if-match conditions
activationcontents: readContext preparation, content sanitization, lock file validation
agentRead-onlyExecute AI engine with tools and MCP servers
detectioncontents: readAI-powered threat analysis — the security gate
safe_outputsScoped writeCreate issues, add comments, open PRs via GitHub API
conclusionWriteAggregate results, generate summary, handle failures
🔒 Why Separate Jobs? Trust boundaries — agent has read-only; safe outputs have write. Immutable job permissions prevent combining. Data flows via GitHub Actions artifacts, preventing output tampering on shared filesystems.

Key Architectural Reasons for Separation

🔒 Trust Boundaries

Agent has read-only; safe outputs have write. Can't combine without violating least-privilege principle.

🛡️ Immutable Permissions

GitHub Actions permissions are per-job and cannot change mid-execution. Structural enforcement.

⛔ Hard Gating

Safe outputs only run if detection passes. Ensures runner never starts on failure.

♻️ Always() Semantics

Conclusion job handles upstream failures. Merging would block writes on safe output failure.

📦 Artifact Handoff

Data flows via GitHub Actions artifacts, preventing output tampering possible with shared filesystem.

🏎️ Different Runners

Detection requires ubuntu-latest; safe output jobs can use lightweight ubuntu-slim.

The Never-Sleeping Teammate

Agentic workflows function as persistent team members that continuously monitor your repository and respond to events:

  • Event-driven: Triggered by repository events (PRs, issues, pushes, schedules, comments)
  • Asynchronous: Works in the background without blocking developers
  • Context-aware: Understands the entire codebase, diffs, configurations, and documentation
  • Intent-based: Defined in plain language Markdown, not YAML rules
  • Iterative: Agents iterate through plan → implement → validate → output cycles

Frontmatter Reference

The YAML frontmatter in your workflow .md file defines everything about how the workflow runs.

.github/workflows/my-workflow.md
---
name: "My Workflow"
description: |
  Human-readable description

on:
  issues:
    types: [opened, reopened]
  schedule: daily
  slash_command:
    name: my-command
  workflow_dispatch:

permissions: read-all
network: defaults

tools:
  github:
    toolsets: [repos, issues, pull_requests]
  web-fetch:
  bash: true
  cache-memory: true

safe-outputs:
  create-pull-request:
    draft: true
    labels: [automation]
  create-issue:
    title-prefix: "[bot] "
    labels: [report]
  add-comment:
  add-labels:
    max: 5

engine:
  id: copilot

timeout-minutes: 15
---

Issue Triage Agent

Trigger: issues.opened 
---
description: |
  Intelligent issue triage assistant that processes new and reopened issues.
  Analyzes content, selects labels, detects spam, gathers context.

on:
  issues:
    types: [opened, reopened]
  reaction: eyes

permissions: read-all
network: defaults

safe-outputs:
  add-labels:
    max: 5
  add-comment:

tools:
  web-fetch:
  github:
    toolsets: [issues]
    min-integrity: none

timeout-minutes: 10
---

# Agentic Triage

You're a triage assistant for GitHub issues. Your task is to analyze issue
#${{ github.event.issue.number }} and perform initial triage tasks.

1. Select appropriate labels for the issue from the provided list.

2. Retrieve the issue content using the `get_issue` tool. If the issue is
   obviously spam, add a comment and exit the workflow.

3. Gather additional context:
   - Fetch the list of labels available in this repository
   - Fetch any comments using the `get_issue_comments` tool
   - Find similar issues using the `search_issues` tool

4. Analyze the issue considering type, severity, and user impact.

5. Write notes, debugging strategies, and reproduction steps.

6. Apply labels using the `update_issue` tool.

7. Add an issue comment with your analysis starting with
   "Agentic Issue Triage".

Grumpy Code Reviewer

Trigger: /grumpy slash command 
---
description: Performs critical code review with a focus on edge cases and bugs

on:
  slash_command:
    name: grumpy
    events: [pull_request_comment, pull_request_review_comment]

permissions:
  contents: read
  pull-requests: read

tools:
  cache-memory: true
  github:
    lockdown: true
    toolsets: [pull_requests, repos]

safe-outputs:
  create-pull-request-review-comment:
    max: 5
    side: "RIGHT"
  submit-pull-request-review:
    max: 1
  messages:
    footer: "> Reluctantly reviewed by [{workflow_name}]({run_url})"

timeout-minutes: 10
---

# Grumpy Code Reviewer

You are a grumpy senior developer with 40+ years of experience who has
been reluctantly asked to review code.

## Your Personality
- **Sarcastic and grumpy** - Not mean, but definitely not cheerful
- **Experienced** - You've seen it all and have strong opinions
- **Thorough** - You point out every issue, no matter how small
- **Concise** - Say the minimum words needed to make your point

## Your Mission

### Step 1: Deduplication Check
Use cache memory to check if you've reviewed this PR before.

### Step 2: Fetch Pull Request Details
Get the PR details, files changed, and diff for each changed file.

### Step 3: Analyze the Code
Look for code smells, performance issues, security concerns,
best practices violations, poor naming, duplicated code.

### Step 4: Write Review Comments
Be specific about file, line number, and what's wrong.

### Step 5: Submit the Review
- APPROVE when no issues need fixing
- REQUEST_CHANGES when issues must be fixed
- COMMENT for non-blocking observations

Daily Repo Status Report

Trigger: schedule daily 
---
description: |
  Creates daily repo status reports. Gathers recent activity
  and generates engaging GitHub issues with productivity insights.

on:
  schedule: daily
  workflow_dispatch:

permissions:
  contents: read
  issues: read
  pull-requests: read

network: defaults

tools:
  github:
    lockdown: false
    min-integrity: none

safe-outputs:
  mentions: false
  allowed-github-references: []
  create-issue:
    title-prefix: "[repo-status] "
    labels: [report, daily-status]
    close-older-issues: true
---

# Daily Repo Status

Create an upbeat daily status report as a GitHub issue.

## What to include
- Recent repository activity (issues, PRs, discussions, releases)
- Progress tracking, goal reminders and highlights
- Project status and recommendations
- Actionable next steps for maintainers

## Style
- Be positive, encouraging, and helpful
- Use emojis moderately for engagement
- Keep it concise - adjust length based on actual activity

## Process
1. Gather recent activity from the repository
2. Study the repository, its issues and its pull requests
3. Create a new GitHub issue with your findings

Documentation Updater

Trigger: push to main 
---
description: |
  Keeps docs synchronized with code changes. Creates draft PRs.

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions: read-all
network: defaults

safe-outputs:
  create-pull-request:
    draft: true
    protected-files: fallback-to-issue
    labels: [automation, documentation]

tools:
  github:
    toolsets: [all]
  web-fetch:
  bash: true

timeout-minutes: 15
---

# Update Docs

You are an Autonomous Technical Writer for ${{ github.repository }}.

### Mission
Ensure every code-level change is mirrored by clear, accurate documentation.

### Voice and Tone
- Precise, concise, developer-friendly
- Active voice, plain English, progressive disclosure

### Workflow
1. **Analyze Repository Changes** - Examine the diff for changed entities
2. **Documentation Assessment** - Review against Diataxis framework
3. **Create or Update Documentation** - Use Markdown format
4. **Create Draft Pull Requests** - Focused, reviewable PRs

> NOTE: Never make direct pushes to the default branch.
> NOTE: Treat documentation gaps like failing tests.

CI Doctor (Failure Analysis)

Trigger: workflow_run.completed 
---
description: |
  Automated CI failure investigator. Deep analysis of Actions failures.

on:
  workflow_run:
    workflows: ["CI Test Suite", "Build Pipeline"]
    types: [completed]
    branches: [main]

if: ${{ github.event.workflow_run.conclusion == 'failure' }}

permissions: read-all
network: defaults

safe-outputs:
  create-issue:
    title-prefix: "[CI Doctor] "
    labels: [automation, ci-failure]
  add-comment:

tools:
  cache-memory: true
  web-fetch:

timeout-minutes: 10
---

# CI Failure Doctor

You are the CI Failure Doctor. Analyze failed GitHub Actions workflows.

## Context
- **Workflow Run**: ${{ github.event.workflow_run.id }}
- **Conclusion**: ${{ github.event.workflow_run.conclusion }}

## Investigation Protocol

### Phase 1: Initial Triage
1. Verify the workflow conclusion is 'failure'
2. Check deduplication cache
3. Get workflow details via `get_workflow_run`
4. List jobs via `list_workflow_jobs`

### Phase 2: Deep Log Analysis
1. Retrieve logs with `get_job_logs` (failed_only=true)
2. Analyze for: error messages, stack traces, dependency failures

### Phase 3: Historical Context
1. Search investigation history in cache memory
2. Search existing issues for related problems

### Phase 4: Root Cause Investigation
Categorize: Code Issues | Infrastructure | Dependencies | Flaky Tests

### Phase 5: Reporting
Create issue with Executive Summary, Root Cause Analysis,
Reproduction Steps, and Recommended Actions.

---

# PR Fix

You are an AI assistant specialized in fixing pull requests with
failing CI checks for PR #${{ github.event.issue.number }}.

1. Read the pull request and comments
2. Take heed of instructions from the triggering comment
3. Check out the branch and set up the development environment
4. Formulate a plan to fix the issue
5. Implement the changes
6. Run tests to verify the fix
7. Run code formatters/linters
8. Push changes to the PR branch
9. Add a comment summarizing the changes

Daily Malicious Code Scan

Trigger: schedule daily 
---
description: Daily security scan for suspicious code patterns

on:
  schedule: daily
  workflow_dispatch:

permissions:
  contents: read
  actions: read
  security-events: read

tracker-id: malicious-code-scan

tools:
  github:
    toolsets: [repos, code_security]
  bash: true

safe-outputs:
  create-code-scanning-alert:
    driver: "Malicious Code Scanner"
  threat-detection: false
---

# Daily Malicious Code Scanner

Review all code changes from the last three days and identify:
- Attempts to exfiltrate secrets or sensitive data
- Code that doesn't fit the project's normal context
- Unusual network activity or data transfers
- Suspicious system commands or file operations
- Hidden backdoors or obfuscated code

When suspicious patterns are detected, generate code-scanning alerts
for visibility in the GitHub Security tab.

## Threat Scoring (0-10)
- Critical (9-10): Active backdoors, exfiltration
- High (7-8): Suspicious with high confidence
- Medium (5-6): Warrants investigation
- Low (3-4): Minor anomalies

Stale Issue Cleaner

Trigger: schedule weekly 
---
description: Identifies and manages stale issues that need attention.

on:
  schedule: weekly
  workflow_dispatch:

permissions:
  contents: read
  issues: read

tools:
  github:
    toolsets: [issues]

safe-outputs:
  add-comment:
  add-labels:
    allowed: [stale, needs-attention]
  close-issue:

timeout-minutes: 10
---

# Stale Issue Cleaner

Review all open issues and identify those that are stale.

## Process
1. List all open issues, sorted by last updated
2. For issues with no activity in 60+ days:
   - If the issue has a linked PR, skip it
   - If filed by a maintainer, add "needs-attention" label
   - Otherwise, add a polite comment asking if still relevant
   - Add the "stale" label
3. For issues marked "stale" with no response in 14+ days:
   - Add a final comment and close the issue
4. Generate a summary of actions taken

gh extension install github/gh-aw

- uses: github/gh-aw/actions/setup-cli@main
  with:
    version: v0.37.18

Core Commands

gh aw init

Initialize repository — configures .gitattributes, creates dispatcher agent, selects engine, configures secrets.

gh aw compile

Compile .md to .lock.yml — schema validation, SHA pinning, security scanning.

gh aw add-wizard

Add pre-built workflow from githubnext/agentics. Handles engine selection and secrets.

gh aw logs

View workflow execution logs. Supports --follow for real-time streaming.

gh aw audit

Audit workflows — run counts, success rates, token usage. Supports --json.

gh aw validate

Validate workflow syntax without compiling. Quick schema and expression checks.

gh aw test

Test workflow locally (preview). Simulate triggers with custom prompts.

gh aw upgrade

Update the CLI to the latest version via gh extension upgrade gh-aw.

EVOLUTION

The DevOps Evolution

From manual processes to autonomous AI teammates — where agentic workflows fit in the journey.

1
Traditional DevOps (2000s)
Cultural & organizational philosophy. Manual processes, no AI required.
2
CI/CD Pipelines (2010s)
Automated build, test, deploy. Deterministic rules — tests pass or fail.
3
AI-Assisted (2021-2024)
Code completion (Copilot), suggestions, chat-based help. Individual developer productivity.
4
Continuous AI (2025+)
AI agents integrated into CI/CD loops. Judgment-based automation — triage, review, documentation, refactoring. This is where gh-aw lives.
5
Autonomous (Future)
Fully autonomous agents that plan, implement, test, and deploy. Humans focus on strategy and architecture.
Key Insight: Agentic DevOps puts AI inside your pipeline. DevOps for Agents wraps your pipeline around AI. GitHub Agentic Workflows is where all directions converge.

QUICKSTART

Getting Started

From zero to your first agentic workflow in five steps.

1

Install the CLI

gh extension install github/gh-aw
gh aw version
2

Initialize Your Repository

cd your-repo
gh aw init

Configures .gitattributes, creates dispatcher agent, selects engine, sets up secrets.

3

Set Up Secrets

EngineSecretDefault
GitHub CopilotCOPILOT_GITHUB_TOKEN✅ Yes
Claude (Anthropic)ANTHROPIC_API_KEYNo
Codex (OpenAI)OPENAI_API_KEYNo
 
gh secret set COPILOT_GITHUB_TOKEN
4

Add Your First Workflow

# Option A: Use the add wizard (fastest)
gh aw add-wizard githubnext/agentics/daily-repo-status

# Option B: Create manually
# Create .github/workflows/issue-triage.md with frontmatter + instructions
5

Compile, Commit & Push

gh aw compile .github/workflows/*.md
git add .github/workflows/
git commit -m "Add agentic workflows"
git push
💡 Best Practices:
  • Start with read-only workflows (reporting, triage) before write workflows
  • Use strict: true (default) for enhanced security validation
  • Enable threat detection for all workflows with safe-outputs
  • Review agent outputs before expanding permissions
  • Use stop-after for cost control during evaluation
DEFENSE IN DEPTH

Three-Layer Security Architecture

How gh-aw ensures agents can't go rogue — structural security from hardware to plan execution.

Layer 1: Substrate Trust

Hardware, Kernel & Container isolation

  • • Memory isolation — each container in its own namespace
  • • CPU & resource isolation via cgroups
  • • Docker networking & iptables boundaries
  • • Zero-Secret Agent Design: Agent NEVER holds auth tokens
  • • LLM tokens in isolated API proxy container
  • • MCP credentials in separate MCP server containers

Layer 2: Configuration Trust

Declarative specification & compilation

  • • Schema validation — frontmatter against JSON schema
  • • Expression safety — only allowlisted Actions expressions
  • • Action SHA pinning — every action locked to commit SHA
  • • Security scanners — actionlint, zizmor, poutine
  • • Strict mode — no write permissions, explicit network config
  • • The gh aw compile command is the trusted compiler

Layer 3: Plan Trust

Staged execution & output validation

  • • Safe Outputs MCP — agent proposes writes as structured artifacts
  • • Call filtering — MCP Gateway validates tool invocations
  • • Secret redaction — masking known secret values
  • • Threat detection — AI-powered analysis of all outputs
  • • Content moderation — policy violation checks
  • • Each stage has defined permissions and artifact handoff

Agent Workflow Firewall (AWF)

The AWF provides network-level isolation for the agent container:

1Creates private Docker network (172.30.0.0/16)
2Binds agent container to that network
3Routes all HTTP/HTTPS through Squid proxy
4Enforces domain allowlist from frontmatter
5Drops iptables capabilities before launch
6Optional chroot mode for full host access control

Network Configuration

network:
  firewall: true
  allowed:
    - defaults  # GitHub, npm, PyPI, Go, Crates, Maven, Docker
    - "api.example.com"
    - "*.trusted-vendor.com"

defaults includes: api.github.com, github.com, python.org, pypi.org, npm registry, golang.org, crates.io, maven.org, docker.io, and common CDNs.

API Proxy Architecture

(no token) → http → API Proxy

🚨 Threat Model Assumption: The system assumes the agent will attempt to: access or corrupt memory of other components, communicate over unintended channels, abuse legitimate channels for unintended actions, confuse control logic, and attempt prompt injection to leak data or execute unauthorized commands.
PATTERNS

The Autonomous Repair Loop

The most powerful pattern: agentic workflows and coding agents working together in a self-healing cycle.

1
CI Build Fails

An agentic workflow detects the failure via workflow_run trigger

2
Agent Analyzes Logs

The CI Doctor reads build logs, identifies root cause, and creates an issue with context

3
Issue Assigned to Copilot

The issue is assigned to the Copilot coding agent for automated remediation

4
Fix PR Created

The coding agent creates a branch, writes the fix, runs tests, and opens a PR

5
Quality Validation

Another agentic workflow validates the PR meets quality gates. Human reviews and merges.

🔄 Key Insight: This creates a fully autonomous repair loop with human review at the PR merge step. The CI Doctor achieves a 69% merge rate on proposed fix PRs in the github/gh-aw repository itself.

Design Patterns for Agentic Workflows

💬 ChatOps

Slash commands in comments: /triage, /analyze, /grumpy. On-demand agent execution triggered by developers.

📅 DailyOps

Scheduled daily maintenance: status reports, security scans, stale issue cleanup, coverage monitoring.

📋 IssueOps

Issue-driven automation: triage on creation, assignment routing, duplicate detection, priority assessment.

🏗️ ProjectOps

Project management automation: sprint reports, velocity tracking, changelog generation, release notes.

🌐 MultiRepoOps

Cross-repository coordination: shared policies, fleet management, organization-wide standards.

🎼 Orchestration

Coordinating multiple worker workflows toward shared goals with Agent HQ governance.

Mental Model: If repetitive work in a repository can be described in words, it is a good candidate for an agentic workflow.
MATURITY

Continuous AI Maturity Model

Five levels of AI integration maturity — assess where your organization stands today.

1
Level 1: Assisted — AI as Autocomplete

Individual developers use Copilot for code completion. No organizational adoption. Ad hoc usage.

2
Level 2: Integrated — AI in the IDE

Teams standardize on AI tools. Custom instructions via .github/copilot-instructions.md. AI-powered PR summaries.

3
Level 3: Automated — AI in CI/CD

AI agents run in Actions workflows. Automated triage, testing, documentation. Human review required for all outputs.

4
Level 4: Orchestrated — Multi-Agent Coordination

Multiple specialized agents work together. Agent HQ for governance. Cross-repo automation. Metrics-driven optimization.

5
Level 5: Autonomous — AI as Teammate

Agents independently plan, implement, test, and propose changes. Humans focus on strategy, architecture, and review. Continuous improvement loops.

Agent Runtime Environment

Every agent runs in an ephemeral, sandboxed environment with pre-installed tools and strict isolation.

Infrastructure

Resources4 vCPUs, 16 GB RAM, 14 GB SSD
Max Job Time6 hours
Setup PhaseMax 59 minutes
Larger RunnersUp to 64 vCPUs, 128 GB RAM, 512 GB disk
Self-HostedSupported with ARC (Actions Runner Controller)

Pre-Installed Tools

Languages: Node.js, Python, Java, .NET, Go, Ruby, Rust, PHP

Build Tools: npm, yarn, pnpm, pip, poetry, Maven, Gradle, CMake, Make, GCC, Clang

Version Control: Git (credential helpers pre-configured)

Utilities: curl, wget, jq, GitHub CLI (gh), zip/unzip, tar

Databases: SQLite pre-installed; others via services key

Ephemeral by Design

A fresh environment is created for each task and destroyed when complete. Key execution phases in the agent job:

1. Repository checkout
2. Runtime setup (Node.js, Python, etc.)
3. Cache restoration from previous runs
4. MCP server initialization
5. Prompt generation (markdown + context)
6. Engine execution (Copilot/Claude/Codex)
7. Secret redaction (mask known secrets)
8. Output upload to Actions artifacts
9. Cache persistence for next run
AUTHORING

Agentic Authoring — AI Writing AI Workflows

Use AI agents to create agentic workflows themselves. Meta-automation at its finest.

Create Workflows with Copilot Agent Mode

Open your repository in VS Code, start Agent Mode, and prompt:

Create a workflow for GitHub Agentic Workflows using
https://raw.githubusercontent.com/github/gh-aw/main/create.md

The purpose is to triage new issues: label them by type
and priority, identify duplicates, ask clarifying questions
when unclear, and assign them to the right team members.

The agent writes the Markdown file, validates the frontmatter, and sets up the workflow.

Creating Workflows from GitHub.com

Open Copilot Chat on github.com and reference the creation template URL in your prompt. The agent will:

1Read the creation template with best practices
2Generate a .md workflow file with proper frontmatter
3Validate the schema and expression safety
4Compile to .lock.yml
5Commit and push to the repository

Repository Structure

After setting up agentic workflows, your .github/ directory looks like this:

.github/
├── workflows/
│   ├── issue-triage.md          # Workflow definition (Markdown)
│   ├── issue-triage.lock.yml    # Compiled workflow (generated)
│   ├── daily-status.md          # Another workflow
│   ├── daily-status.lock.yml    # Its compiled output
│   └── copilot-setup-steps.yml  # Coding agent setup
├── copilot-instructions.md      # Custom instructions
├── agents/
│   └── agentic-workflows.agent.md  # Dispatcher agent
├── hooks/
│   └── security.json            # Copilot hooks
└── instructions/
    └── typescript.instructions.md  # Path-specific instructions
