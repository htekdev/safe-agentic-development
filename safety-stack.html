Layer 1

Custom Instructions — Soft Guardrails

What It Is

Custom instructions are Markdown files that define how AI agents should behave in your repository. They specify coding standards, technology choices, testing requirements, security rules, and architectural boundaries — all in natural language that the agent reads at the start of every session.

Why It Matters for Safety

Instructions are the first line of defense — the "constitution" that shapes every decision the agent makes. Without them, agents guess at your team's conventions, potentially choosing wrong frameworks, skipping tests, or violating architectural boundaries. With them, every session starts with your team's standards already loaded.

Key Features

  • Repository-wide rules via .github/copilot-instructions.md — automatically applied to every session
  • Path-specific rules via .github/instructions/*.instructions.md — scoped to directories
  • Organization-level instructions for enterprise-wide standards
  • Agent instructions via AGENTS.md — compatible across Copilot, Claude, Gemini
  • Auto-generation — Copilot can generate its own instructions by analyzing your codebase
  • Version controlled — lives in the repo like code; reviewable, auditable, evolvable
⚠️ Limitation: Instructions are soft constraints — agents may not always follow them perfectly. That's why the next layer exists.

Code Example

<!-- .github/copilot-instructions.md -->
# Project Standards

## Tech Stack
- TypeScript with strict mode enabled
- React 18 with functional components and hooks
- PostgreSQL with Prisma ORM
- Vitest for unit tests, Playwright for E2E

## Security
- Never hardcode API keys, tokens, or secrets
- Use environment variables for all configuration
- All user input must be validated and sanitized
- Use parameterized queries — no raw SQL

## Testing
- Every source change requires corresponding tests
- Minimum 80% coverage on all modules
- Run `npm test` before committing

Layer 2

Agent Hooks & Extensions — Hard Guardrails

What It Is

Hooks and extensions are programmatic governance gates that intercept agent actions in real time. Unlike instructions (which are advisory), hooks are deterministic — they execute your code at specific lifecycle points with guaranteed outcomes. They can block destructive commands before they run, detect secrets before they're written to files, and enforce testing requirements before commits happen.

Why It Matters for Safety

Instructions tell the agent what to do. Hooks enforce what the agent cannot do. When an agent tries to run rm -rf /, a hook blocks it — instantly, deterministically, every time. When it tries to commit code without tests, a hook denies the commit. This is the layer that turns guidelines into guardrails.

Key Features

  • Pre-tool hooks (onPreToolUse) — intercept and block actions before execution
  • Post-tool hooks (onPostToolUse) — inject feedback after actions for self-correction
  • Session hooks (onSessionStart) — inject baseline security context
  • Shell Hooks (.github/hooks/) — JSON + shell scripts
  • CLI Extensions (.github/extensions/) — Full Node.js with stateful tracking
  • Hookflows (.github/hookflows/) — Declarative YAML governance rules
  • Zero installation — auto-discovered and loaded; no npm install needed
  • Hot reload — extensions reload mid-session

Three Foundational Patterns

  • Test Enforcer — blocks commits without corresponding test changes
  • Lint-on-Edit — runs linter after every file edit, feeds results back
  • Repo Shield — blocks rm -rf /, DROP DATABASE, force push; detects hardcoded secrets

Code Example — Test Enforcer Extension

// .github/extensions/test-enforcer/extension.mjs
import { approveAll } from "@github/copilot-sdk";
import { joinSession } from "@github/copilot-sdk/extension";

const modifiedSourceFiles = new Set();
const modifiedTestFiles = new Set();

const session = await joinSession({
  onPermissionRequest: approveAll,
  hooks: {
    onPreToolUse: async (input) => {
      if (input.toolName !== "powershell") return;
      const cmd = String(input.toolArgs?.command || "");
      if (!/\bgit\b.*\bcommit\b/.test(cmd)) return;

      const untestedFiles = [...modifiedSourceFiles].filter((src) => {
        const base = src.replace(/\.[^.]+$/, "");
        return ![...modifiedTestFiles].some((t) => t.includes(base));
      });

      if (untestedFiles.length > 0) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason:
            `[test-enforcer] BLOCKED: Source files modified without tests.`,
        };
      }
    },
  },
  tools: [],
});

Layer 3

Planning Mode — Structured Approach

What It Is

Planning Mode is a built-in "plan-before-code" capability across VS Code, Visual Studio, JetBrains, and the Copilot CLI. Instead of immediately generating code from a vague prompt, Copilot first creates a structured implementation plan that you review, refine, and approve before any code is written. Spec-Driven Development (SDD) extends this with a full methodology.

Why It Matters for Safety

A vague prompt like "add photo sharing to my app" forces the model to guess at thousands of unstated requirements. Planning Mode forces clarity upfront, converting ambiguous intent into explicit, reviewable steps. This prevents the most common failure mode of agentic development: agents confidently building the wrong thing at speed.

Key Features

  • Plan agent (/plan command) — analyzes codebase, asks clarifying questions, produces structured plans
  • Session persistence — plans saved to plan.md, surviving context compaction
  • Spec Kit — open-source toolkit from GitHub for full spec-driven development
  • Four-phase workflow: Specify → Plan → Tasks → Implement
  • Cross-IDE support — VS Code, Visual Studio, JetBrains, Eclipse, Xcode, Copilot CLI
  • Background execution — implement the plan in a background session

SDD Artifact Stack

  • constitution.md → Non-negotiable project principles
  • spec.md → What & Why (user stories, acceptance criteria)
  • plan.md → How (architecture, data models, constraints)
  • tasks.md → Executable work (phased, dependency-ordered)

Code Example — Planning Prompt

/plan Implement user profile editing for our Express.js API.

Requirements:
- Authenticated users can update name, bio, and avatar
- Avatar upload limited to 2MB, JPEG/PNG only
- Email changes require verification
- All changes audited in activity log
- Must work with existing JWT auth in src/middleware/auth.ts
- Follow repository pattern in src/repositories/
- Include integration tests using our Vitest setup

Layer 4

Model Context Protocol (MCP) — Controlled Tool Access

What It Is

MCP is an open standard created by Anthropic that gives AI agents a standardized way to connect to external tools, databases, and services — like a "USB-C port for AI." Instead of custom integrations for every data source, MCP provides a single protocol that works across Claude, Copilot, ChatGPT, Cursor, and other clients.

Why It Matters for Safety

MCP provides controlled expansion of agent capabilities. Without MCP, teams might give agents overly broad access to external systems. With MCP, every tool is registered with explicit schemas, permissions, and audit trails. Each MCP tool call in Copilot requires explicit user approval, creating a natural safety checkpoint.

Key Features

  • Dynamic capability discovery — agents learn available tools at runtime
  • Permission boundaries — configure read-only modes, tool whitelisting, per-user permissions
  • GitHub's Remote MCP — managed, OAuth-authenticated, auto-upgrading endpoint
  • Default servers for the coding agent: GitHub MCP (read-only) and Playwright (localhost only)
  • 10,000+ community servers across registries (Smithery, MCP.SO, Glama)
  • Multi-language SDKs — TypeScript, Python, C#, Java, Kotlin, Ruby, Swift
  • Enterprise MCP gateways — centralized management with Docker, WSO2, Lunar.dev

Code Example — MCP Configuration

// ~/.copilot/mcp-config.json
{
  "servers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${POSTGRES_CONNECTION_STRING}"
      }
    },
    "docs": {
      "command": "npx",
      "args": ["-y", "context7-mcp"]
    }
  }
}

Layer 5

Agentic Workflows — Automated CI/CD

What It Is

Agentic Workflows bring AI coding agents into GitHub Actions with built-in, enterprise-grade guardrails. They are Markdown files with YAML frontmatter that define what an agent should do, when it should run, and — critically — what it's allowed to write. They compile to standard GitHub Actions via gh aw compile.

Why It Matters for Safety

Agentic Workflows are the most security-hardened layer of the stack. They run in sandboxed containers with zero access to secrets, firewalled internet, and kernel-enforced isolation. All write operations are staged through a "safe outputs" pipeline that specifies, limits, and sanitizes every action the agent can take.

Key Features

  • Read-only by default — agents can read everything but write nothing unless authorized
  • Safe outputs — pre-approved write operations with quantity limits and content sanitization
  • Zero-secret agents — agents have no access to secrets; LLM tokens held by a separate API proxy
  • Sandboxed execution — dedicated containers with chroot jail, firewalled egress, tmpfs overlays
  • Intent-driven — write workflows in natural language, not YAML scripts
  • Multi-engine — supports Copilot, Claude Code, and OpenAI Codex
  • Comprehensive logging — network activity, model requests, tool invocations all logged

Code Example — Issue Triage Workflow

<!-- .github/workflows/issue-triage.md -->
---
on:
  issues:
    types: [opened]
permissions:
  issues: write
tools:
  github:
    toolsets: [issues]
safe-outputs:
  add-label:
    max: 5
  add-comment:
    max: 1
---

# Issue Triage Agent

When a new issue is opened:
1. Read the issue title and body
2. Classify as: bug, feature request, question, or documentation
3. Apply the appropriate label
4. If the issue is unclear, leave a polite comment asking for details
5. If it appears to be a duplicate, note the related issue number

Layer 6

Human-in-the-Loop — The Final Gate

What It Is

Human-in-the-Loop (HITL) is the design pattern ensuring that humans approve every critical agent action before execution. Agents create pull requests, but never merge them. Agents propose changes, but humans make every irreversible decision. The agentic loop maintains human control through structured processes at every checkpoint.

Why It Matters for Safety

No matter how sophisticated the automated layers above, AI will make mistakes. HITL is the backstop that prevents mistakes from becoming incidents. By requiring human approval for all irreversible actions — merges, deployments, CI triggers — the blast radius of any agent error is contained to a review-ready PR, not production.

Key Checkpoints

  • Agents create PRs, never commit directly to protected branches
  • PRs do NOT auto-run CI — humans must manually trigger Actions
  • MCP tool calls require explicit approval
  • Code review is always required before merge
  • Enterprise admins can enforce additional approval gates

The Agentic Loop

  • Recon — Agent analyzes the codebase to understand context
  • Reproduce — Agent identifies the problem (failing test, reported bug)
  • Localize — Agent finds the relevant code
  • Repair — Agent implements the fix
  • Validate — Agent runs tests and security checks
  • Explain — Agent documents what it did and why
  • Merge — Human reviews and approves

What Arrives for Human Review

The safety stack doesn't remove humans — it augments them. Every PR still requires human review. But the code arriving for review has already been:

✅ Tested             — test-enforcer ensured tests were written
✅ Linted             — lint-on-edit ran the linter after every edit
✅ Scanned for secrets — repo-shield + secret scanning checked
✅ Analyzed for vulns  — CodeQL static analysis completed
✅ Dependency checked  — GitHub Advisory Database scanned
✅ Planned & reviewed  — implementation followed an approved plan

This is more thorough than any manual process.

Source: Xebia — "Agentic Development"

GitHub's Six Rules for Agentic Products

The official security principles that govern every AI agent feature GitHub builds — designed to balance capability with safety.

1

All Context Is Visible

Remove invisible Unicode directives, hidden HTML tags, and masked information before passing to the agent. Only information clearly visible to maintainers reaches the agent — preventing prompt injection attacks through hidden context.

2

Firewall the Agent

Apply network firewalls to limit external resource access. Users configure allowed network connections and block unwanted ones. Generated code is presented as preview, never auto-executed. MCP interactions bypass the firewall as a controlled exception.

3

Limit Sensitive Data Access

Only provide information absolutely necessary for the agent to function. CI secrets and files outside the current repository are never passed to agents. GitHub tokens are revoked once the agent session completes.

4

Prevent Irreversible Changes

Agents cannot initiate irreversible state changes without human approval. Copilot creates pull requests — never commits directly to default branches. PRs don't auto-run CI. MCP tool calls ask for approval before executing.

5

Attribute Actions Clearly

Every agentic interaction is attributed to both the initiating user and the agent. Pull requests show co-commits, and the Copilot identity makes AI-generation clearly visible — enabling accountability and incident investigation.

6

Authorized Users Only

Agents gather context only from authorized users and always operate under permissions granted by the initiating user. For public repos, only reads issue comments from users with write access as an additional security layer.

Source: GitHub Blog — Rahul Zhade, "How GitHub's agentic security principles make our AI agents as secure as possible" (November 2025)

Defense-in-Depth Philosophy

No single layer is enough. Each layer catches what the layer above misses. Together, they create a security posture that exceeds what any individual control could achieve alone.

🧅 Independent Layers

Each layer operates independently with its own enforcement mechanism. If instructions fail to prevent a bad pattern, hooks block it. If hooks miss something, the workflow sandbox contains it. If the sandbox fails, humans catch it in review.

🔄 Soft → Hard Gradient

The stack moves from soft guidance (instructions) through programmatic enforcement (hooks) to environmental isolation (sandboxes) to final human judgment. Each layer is progressively more restrictive and harder to bypass.

🎯 No Single Point of Failure

Unlike a single "AI safety switch," the stack has no single point of failure. An attacker would need to defeat instructions AND hooks AND planning AND MCP controls AND workflow sandboxing AND human review — simultaneously.

⚡ Speed Without Sacrifice

The paradox: adding more safety layers actually increases velocity. Teams ship faster because they're not manually checking everything. The automated layers handle the "did you remember to…" checklist, freeing humans for creative work.

📊 More Thorough Than Manual

Manual code reviews catch ~60-70% of issues at best. The safety stack checks for secrets, runs linters, executes tests, performs static analysis, scans dependencies, and validates plans — every time, without fatigue.

🏢 Enterprise-Grade Governance

The Agent Control Plane provides organization-wide policy enforcement, centralized audit trails, session analytics, and compliance reporting — extending defense-in-depth from individual repos to the entire organization.

A strict hierarchy ensures that higher-level policies always take precedence. Agents are the most constrained.
